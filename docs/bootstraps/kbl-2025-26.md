# KBL 2025-26 Bootstrap

Methodology template for league-level + sub-league team-coverage bootstraps. KBL (Korean Basketball League) is the pilot for the 5-sport zero-coverage cohort surfaced in `PROJECT_STATE.md` 2026-05-17: Handball, Snooker, Volleyball, Rugby League, Golf.

This document describes the workflow followed during the KBL bootstrap so future bootstraps in the cohort can mirror it. It is intentionally generic where the steps apply across leagues; KBL-specific commentary is called out as such.

> **Two sections in this doc.** The **Workflow** section (§1-8) covers KBL bootstrap methodology — seed format, script structure, tests, apply runbook. The **Methodological observations** section (Patterns A-G) is a **general diagnostic-patterns reference**, not KBL-specific. Patterns were discovered during KBL + subsequent Phase 2 workstreams and documented here because the KBL pilot was the origin point. When the pattern set stabilizes, these may extract to a standalone `docs/methodology/diagnostic-patterns.md` — for now they live here to avoid premature doc proliferation.

## Workflow

### 1. Scope doc (operator-driven)

Operator writes a scope doc covering:

- Goal statement (what coverage shipping means)
- Source-of-truth selection (which authoritative source for canonical naming + sponsor variants)
- Out-of-scope clarifications (player rosters, women's leagues, lower divisions, predecessor teams)
- Manifest format intent (per-team rows, alias coverage, country code)
- Workflow steps (triplet files: seed + bootstrap + tests, plus optional Makefile target + docs)
- Operator role expectations (review checkpoints, time budget)
- Success criteria + risks + mitigations

Scope doc commits to a Track 2 work order. KBL scope doc lived in conversation rather than a committed file; future bootstraps may benefit from committing the scope doc to `docs/bootstraps/<league>-scope.md` before code starts.

### 2. Operator pre-execution discovery queries (operator-side)

Before any code lands, operator runs production discovery queries to characterize the data landscape. For KBL these were:

- **Production variant capture** — `SELECT … FROM sp.review_queue WHERE provider_record_id LIKE '<LEAGUE_PREFIX>%'`. Captures the team-name variants actually appearing in production records. Tells the manifest writer what aliases real records use.
- **Existing-team cross-check** — `SELECT … FROM sp.teams WHERE country_code = '<KOR>' OR canonical_name LIKE '%<known team substring>%'`. Discovers whether the legacy 2A.5 bootstrap already seeded any rows (which would route the bootstrap to UPDATE-branch instead of INSERT-branch).
- **Cross-league collision search** — variant of the cross-check expanded with bare-substring LIKE patterns on team-name fragments. Surfaces non-target-league teams whose names happen to share substrings with the target (e.g., Egis Körmend in Hungary sharing "Egis" with KBL's KCC Egis; Cimarrones de Sonora sharing "Sono" with KBL's Goyang Sono). Drives the alias-distinctiveness constraint.
- **Historical-variant capture via `sp.resolution_log`** (added 2026-05-19 as a mandatory pre-merge step — see methodological observation below). DISTINCT on `home_provider_normalized` / `away_provider_normalized` across all historical records for the target league's provider_record_id pattern. Surfaces empirical team-form variants the provider has actually used in production.

Operator shares query results with the seed-write process. Code execution does not begin until the data landscape is characterized.

### 2a. Pre-merge production-data verification (mandatory)

**Added as a mandatory step after the KBL pilot caught a real alias gap at this step that authoritative-source curation missed.**

Before merging the bootstrap PR, operator runs:

```sql
SELECT DISTINCT team_form FROM (
  SELECT reason_detail->>'home_provider_normalized' AS team_form
  FROM sp.resolution_log
  WHERE provider = '<provider>'
    AND reason_detail->>'sport' = '<sport>'
    AND provider_record_id LIKE '<league-prefix>%'
  UNION
  SELECT reason_detail->>'away_provider_normalized'
  FROM sp.resolution_log
  WHERE provider = '<provider>'
    AND reason_detail->>'sport' = '<sport>'
    AND provider_record_id LIKE '<league-prefix>%'
) AS forms
WHERE team_form IS NOT NULL ORDER BY team_form;
```

For every distinct team_form returned, verify that the seed manifest has an alias covering it. If not, add the missing alias before merge — speculative authoritative-source curation (NamuWiki, Wikipedia, official league site) does not always capture the provider-specific naming conventions. **The provider's actual generated form trumps the team's official name** for alias coverage purposes.

The KBL pilot found that Kalshi uses the 2-token "City Nickname" form ("Goyang Skygunners") in KBL ticker generation, dropping the sponsor token "Sono" that appears in the official name "Goyang Sono Skygunners." Without the pre-merge verification, the bootstrap would have shipped with 0% match rate on the actual production records.

### 3. Operator decisions on framing questions

Several framing questions emerged during the KBL bootstrap that recur across league bootstraps. Documenting them here so future bootstraps can reuse the decision rationale:

- **F1: Canonical_name policy when an existing legacy row has the team but with a different short-form canonical.**
  - **KBL decision: A (mirror PR #156 precedent).** Keep the existing canonical; add the current 2025-26 official form as an alias on the same row. Rationale: matches PR #156 (national-teams bootstrap) precedent exactly; FL is the §9.3 authoritative source for canonical_name in the automatic ingestion flow; UPDATE-ing canonicals would create drift between bootstrap and FL.
  - Future bootstraps should default to the same policy unless a specific architectural argument justifies an UPDATE-branch on canonical_name.

- **F2: Sponsor-rebrand alias coverage.** Teams that recently rebranded (KBL example: Anyang KGC → Anyang JeongKwanJang in 2023) need coverage of both the legacy and current sponsor brand, plus romanization variants of the new brand if Korean. Operator decision was "bracket more rather than less": include all distinct normalized variants of both old and new sponsor brands.

- **F3: Hangul (or other non-Latin script) coverage strategy.** For Korean (and analogous decisions for other non-Latin scripts in future cohorts: Cyrillic, Arabic, etc.). Operator decision was to run a production query first to determine whether records carry Hangul-only home/away values. KBL's query returned 0 Hangul rows, so the seed shipped with partial Hangul coverage (3 of 10 teams confirmed) and a follow-up tracking issue. If the analogous query in a future bootstrap returns Hangul-only records, full coverage becomes a prerequisite before v1.

- **Q3: `sp.team_aliases.source` value.** KBL uses `bootstrap_league_coverage` (new value, not in existing source enum comment). This is a **generic value reused across the 5-sport cohort** (Handball, Snooker, Volleyball, Rugby League, Golf, Darts). Per-bootstrap source values (`bootstrap_kbl`, `bootstrap_handball`, etc.) were considered and rejected — too specific for the analytics use case. The one-label-across-all-bootstraps approach gives future tooling a single WHERE-clause to distinguish bootstrap-seeded aliases from operator-added (`manual_review`, `human_curated`) and provider-discovered (`kalshi`, `fl`, `polymarket`, `oddsapi`).

### 4. Seed file (`scripts/<league>_seed.py`)

Format: list of 4-tuples `(canonical_name, country_code, aliases_tuple, notes)`. Extends PR #156's national-teams 3-tuple format with an aliases dimension.

Key invariants the seed file enforces:

- **Canonical name semantics** match F1 decision: legacy form for UPDATE-branch rows; current official for INSERT-branch rows.
- **Alias distinctiveness:** every alias is multi-token, sponsor-prefixed, or non-Latin script. No bare team-name aliases that would collide cross-league within the same `sport_id`.
- **Country code is uniform** for league bootstraps (KBL: all KOR).
- **Hangul / non-Latin script coverage** is partial-and-tracked when authoritative sources don't cover all teams uniformly.
- **Source-value constant** is exported alongside the manifest so the bootstrap script and tests reference a single source of truth.

The seed file's docstring carries the re-curation runbook: how to update the manifest when league membership, sponsor brands, or city locations change. This makes the file self-documenting for the next operator who needs to maintain it.

### 5. Bootstrap script (`scripts/bootstrap_<league>.py`)

Mirrors `scripts/bootstrap_national_teams.py` structure with two structural extensions:

- **Three-branch classifier on `sp.teams`:**
  - (a) **INSERT new** — manifest entry not present in sp.teams by normalized canonical_name
  - (b) **BACKFILL country_code** — manifest entry present in sp.teams with country_code NULL (Phase 1.5 backfill pattern from PR #156)
  - (c) **SKIP** — manifest entry present with country_code already populated

- **Parallel alias classifier on `sp.team_aliases`:**
  - (a) **INSERT new alias** — manifest alias not present for this team_id under this source
  - (b) **SKIP existing alias** — already present (idempotent re-run)

The alias classifier runs as a second sequential pass after team-row writes resolve. Both writes share a single transaction via `await session.commit()` at the end.

Defensive idempotency at three layers:
1. Python-side dedup in the manifest itself (raw alias forms that normalize to the same string aren't seeded twice).
2. Python-side dedup against existing-state load before writes.
3. Postgres-side `ON CONFLICT DO NOTHING` on the alias's `(alias_normalized, source)` UNIQUE constraint.

Exit codes mirror `bootstrap_national_teams.py`:
- `0` — success (writes OR idempotent no-op)
- `1` — DATABASE_URL unavailable
- `2` — bad CLI args
- `3` — required sport row missing from sp.sports

CLI flags:
- `--dry-run` — classify but don't write; print would-do counts

### 6. Test file (`tests/test_bootstrap_<league>.py`)

Two test classes:

- **`TestManifestShape`** — pure-data tests, run always. Validates the seed manifest against invariants the bootstrap depends on (size, tuple arity, country code uniformity, normalization non-emptiness, distinctiveness constraint, dedup, F1/F2/F3 decisions encoded).

- **`TestKblBootstrapIntegration`** (or analogous per-league name) — `SP_INTEGRATION_DB`-gated. Subprocesses the bootstrap script against a real Postgres and verifies INSERT-branch happy path, BACKFILL UPDATE-branch on pre-seeded legacy rows, idempotency on second run, dry-run mode behavior, and source-value attribution on inserted aliases.

Test surface is roughly 1.5x the PR #156 test surface because the alias-write path needs its own idempotency + source-attribution tests.

### 7. Makefile target

Add a `bootstrap-<league>` target mirroring `bootstrap-national-teams`. Two-line addition to the `.PHONY` line + a help-text line + the target body. Allows the operator-facing apply command to follow the project's standard idiom (`make bootstrap-<league> ARGS="--dry-run"`).

### 8. PR + production-apply sequence

1. Open PR with the triplet files + Makefile addition + this docs file (or follow-on for the cohort). PR description should include dry-run output captured against a Neon dev branch.
2. Operator spot-checks the manifest at PR review — verify team names against authoritative source, sponsor variants reasonable, country code correct, no bare-substring alias risks.
3. Merge.
4. **Verify connection endpoint matches production BEFORE any write.** Per Pattern D below — run `SELECT current_database(), current_schema(), inet_server_addr();` in the psql session that will run the apply. Cross-reference `inet_server_addr()` against the known production endpoint (Neon console → production branch → connection details). If they don't match, the apply would land on the wrong branch. Mechanical 5-second check; prevents an entire class of silent-success-on-wrong-branch errors.
5. Operator runs `make bootstrap-<league> ARGS="--dry-run"` against production. Captures output as a PR verification comment per Issue #129 convention.
6. Operator runs `make bootstrap-<league>` against production. Captures counts.
7. Wait one cron cycle. Verify pending review_queue records for the league route differently than before (`fuzzy_no_team_resemblance` no_match → either strict/alias auto-apply, or asymmetric routing to review_queue with surface-able candidates, depending on what the bootstrap's coverage enables for each record).
8. Capture operational impact (e.g., 24h reduction in pending queue depth for the league) as a follow-up PR comment.

### 9. Tracking issues filed alongside PR

Per the KBL pilot's experience, file the following tracking issues if applicable to the bootstrap:

- **Hangul / non-Latin script follow-up coverage** — if v1 shipped with partial coverage per F3, file a tracking issue covering the remaining teams. Priority is determined by whether production records actually carry non-Latin script for the uncovered teams.
- **`country_code = NULL` data-hygiene observation** — if cross-check queries surfaced country_code=NULL patterns across club-level rows beyond the target league (KBL's Query A2 found this for Mexican baseball teams, Hungarian basketball, etc.), note as a generalized observation in PROJECT_STATE rather than per-issue filing. The pattern resolves naturally as future bootstraps run.

## Methodological observations (Phase 2C cohort patterns)

Two patterns established during the KBL pilot that should carry forward to Handball / Snooker / Volleyball / Rugby League / Golf / Darts bootstraps in the 5-sport cohort.

### Pattern A: Production-data verification via `sp.resolution_log` is mandatory before bootstrap merge

The DISTINCT-team-form query described in step 2a above is **non-optional** for league bootstraps where the league has historical records in `sp.resolution_log`. Authoritative-source curation (NamuWiki, Wikipedia, official league site) captures the team's official name + common variants, but providers may generate non-obvious short forms that don't appear in any authoritative source.

The KBL pilot proved this concretely: the seed-write process, working from Wikipedia and NamuWiki, generated "Goyang Sono Skygunners" (the 2025-26 official full form) as the canonical-augmenting alias. But Kalshi's ticker-generation code uses a 2-token "City Nickname" pattern that omits the sponsor token — every historical KBL Kalshi record in `sp.resolution_log` carried `Goyang Skygunners`, not `Goyang Sono Skygunners`. Without the pre-merge verification step, the bootstrap would have shipped with **0% match rate on the actual production records** — the verification target (3 → 0 KBL pending records) would not have landed.

**The provider's actual generated form trumps the team's official name** for alias coverage purposes. The mandatory pre-merge step is the gate that catches this.

For leagues with **no historical `sp.resolution_log` data** (a future league that's never produced records, e.g., because the provider hasn't covered it yet), the DISTINCT query returns empty. In that case, the bootstrap ships with authoritative-source aliases only; production-data verification becomes a post-deploy continuous-feedback step (operators surfacing missed aliases via review_queue, follow-up `alias_add` PRs filling gaps).

### Pattern B: Observe-then-react over pre-seed-speculate for unproven aliases

For teams within the bootstrapped league that have **no historical records** in the provider's data, do NOT pre-seed speculative short-form aliases. Reasons:

1. **Cross-league collision risk.** Speculative short-forms increase the surface for accidental collisions with non-target-league teams (Egis Körmend / KCC Egis precedent). Without empirical verification that a specific provider uses a specific form, every speculative alias is unverified risk.

2. **§7.7 daily cron re-resolution.** The daily resolver cron retroactively processes pending review_queue records when new aliases are added (verified during Track A priority 3 work — see PROJECT_STATE 2026-05-18). This means aliases added after the bootstrap ships still benefit prior records — there's no "missed window" penalty for incremental coverage.

3. **Real-data feedback over speculation.** When a previously-unseen team starts appearing in provider data, the operator validation cycle surfaces the actual provider-generated form. That empirically-verified form becomes the alias — not the operator's guess at what form the provider might use.

The KBL pilot applied this: 8 of 10 KBL teams had no historical Kalshi records. The bootstrap shipped with their authoritative-source aliases only (canonical + city-prefixed short form where applicable + Hangul where confirmed). When Kalshi starts ingesting records for those 8 teams, operator-validation cycles will surface real variants for follow-up `alias_add` PRs.

**Trade-off accepted:** brief delay between first-Kalshi-record-arrival-for-team-X and full alias coverage for team-X, in exchange for zero pre-seed-collision risk and a discipline of empirically-grounded alias coverage. The §7.7 daily cron narrows the delay window to ~24h (the next cron pass after `alias_add` runs).

### Pattern C: Clear bytecode before invoking diagnostic scripts against production from local Python

When a local Python environment hosts a script that connects to production data (e.g., `scripts/run_resolver_pass.py --run-mode standalone` against a production `DATABASE_URL`), `__pycache__` can outlive source changes. The script's source shows the fix, `grep` confirms the fix, but the script still exhibits pre-fix behavior because Python loaded a stale compiled bytecode from a previous run.

**Symptom shape:** "audit-says-safe but observation-says-crash" — every code path you can find says the bug shouldn't fire, but production-data invocation produces the same error as before the fix.

**Diagnostic order when this shape surfaces:**

1. **Environmental first** — clear bytecode (`find <module-dir> -name "__pycache__" -exec rm -rf {} +`), verify Python's importing the right file (`python -c "import <module>; print(<module>.__file__)"`), confirm venv is the expected one (`which python`).
2. **Code second** — only after environmental causes are ruled out, audit construction sites and invariants.

**Cost asymmetry:** environmental diagnosis is ~30 seconds to test; code-site diagnosis is multiple message exchanges plus potential wrong-site patches. Always run the environmental check first when the bug shape is "audit says one thing, observation says another."

Discovered during PR #171 hotfix diagnostic cycle on 2026-05-19 when stale `.pyc` produced 30 false-positive `ValidationError` "crashes" despite the fix being committed, pushed, merged, and Railway-deployed correctly. Misdiagnosis cost ~2-3 hours of investigation across multiple message exchanges, including a proposal for unnecessary defensive patches at 4 additional code sites. Resolved when stale-bytecode hypothesis was tested by clearing `__pycache__` and re-running the same diagnostic — crash count dropped from 30 to 0 with no other change.

Tracked as Issue #172 for the complementary script-side process improvement (stderr warning + docstring guidance in `scripts/run_resolver_pass.py` for `--run-mode standalone` invocations).

### Pattern D: Verify the connection endpoint matches the expected branch before any database write operation

When a Neon-hosted Postgres setup exposes both production and dev/test branches via different connection strings, the `DATABASE_URL` env var becomes the silent disambiguator between "wrote to production" and "wrote to a branch." Bootstrap scripts and migration runs report success against whatever DB the connection string targets, regardless of intent.

**Symptom shape:** apply script reports clean success (row counts match expectations, no errors), but downstream production behavior shows no change. Verification queries that don't explicitly check the connection endpoint return data consistent with apply NOT having landed on production.

**Pre-flight check (operator-side, ~5 seconds):**

```sql
SELECT current_database(), current_schema(), inet_server_addr();
```

Run before any write-side operation against `DATABASE_URL`. Cross-reference `inet_server_addr()` against the known production endpoint (Neon console → production branch → connection details). If they don't match, the write would land on the wrong branch. This is now step 4 of the production-apply sequence in section 8 above.

**Cost asymmetry:** verification is 5 seconds; recovery from a silent wrong-branch write is hours-to-days of investigation, plus the data-integrity risk if production drifts undetected. Pattern D mirrors Pattern C's environmental-before-code discipline at the write-path layer.

Discovered during KBL bootstrap apply on 2026-05-19 when the apply succeeded against a `bootstrap-test` branch (endpoint `ep-square-wave-akhp46h0`) instead of production (endpoint `ep-fragrant-frog-ak3esp11`). The 2026-05-19 apply reported "8 teams inserted, 2 backfilled, 21 aliases" — all true, but written to bootstrap-test. Production state remained pre-apply. Caught on 2026-05-20 morning when KBL queue-depth verification showed unchanged state and source-value diagnostic showed `bootstrap_league_coverage` had zero rows in production. Recovery: confirm correct endpoint via Neon console, re-set `DATABASE_URL`, re-run `make bootstrap-kbl` against production (idempotent, safe). Total recovery cost: ~30 minutes once the wrong-endpoint hypothesis was identified.

Pattern D's pre-flight check is mechanical and ~5 LOC of operator runbook discipline; the recurrence-prevention value across the remaining 5-sport cohort (Handball, Snooker, Volleyball, Rugby League, Golf) is substantial — each future bootstrap apply has the same silent-wrong-branch risk surface absent the check.

**Sub-pattern — verify-endpoint-before-READ (added 2026-05-20 during Phase 2 Track A scope-doc cycle):**

The same endpoint-verification discipline applies to measurement scripts that READ from production for analytics or diagnostic purposes — not just write-side bootstrap applies. Reasoning:

- A measurement script connected to the wrong DB produces wrong baselines.
- Wrong baselines drift undetected because no one cross-checks them daily.
- A week of measurement against a dev branch would create false trends that drive wrong tuning decisions.

Concrete instance: Phase 2 Track A's `scripts/daily_diff.py` (per PR #175 scope doc §10) runs `SELECT current_database(), current_schema(), inet_server_addr();` at script start. Compares `inet_server_addr()` against a hard-coded production endpoint identifier (or against an env var like `EXPECTED_PRODUCTION_ENDPOINT`). Exits cleanly with an error message if mismatch.

Same pattern, two directions:
- **Write path:** verify-endpoint-before-write prevents silent wrong-branch writes (KBL bootstrap recovery instance).
- **Read path:** verify-endpoint-before-read prevents silent wrong-baseline measurements (Track A measurement-substrate instance).

Don't proliferate Patterns when one extends naturally. Pattern D covers both directions.

**Refinement — platform-aware endpoint signal (added 2026-05-21 during Track A Deliverable 2 Pattern D pre-flight against Neon):**

The original pre-flight SQL above uses `inet_server_addr()` as the endpoint discriminator. Validated empirically against production Neon: `inet_server_addr()` returns `169.254.254.254` — Neon's link-local proxy address, identical across branches of the same Neon project. As a branch discriminator it is useless on Neon.

The real branch discriminator is the `DATABASE_URL` hostname (e.g., `ep-fragrant-frog-ak3esp11.us-east-2.aws.neon.tech`). Refined pre-flight for Neon-hosted deployments:

```sql
-- Still useful: confirms DB name, catches non-Neon targets.
SELECT current_database();
```

```python
# Plus URL-host check (script-side, before opening the SQL session):
from urllib.parse import urlparse
hostname = urlparse(os.environ["DATABASE_URL"]).hostname
assert EXPECTED_PRODUCTION_DB_HOST in hostname  # e.g. 'ep-fragrant-frog-ak3esp11'
```

Concrete instance: `scripts/daily_diff.py:_pattern_d_pre_flight` (Phase 2 Track A Step 4, commit `e7fd82c`) implements this exact pair of checks. Env contract:
- `EXPECTED_PRODUCTION_DB_NAME` — must equal `current_database()` result (default `neondb`)
- `EXPECTED_PRODUCTION_DB_HOST` — substring that must appear in the parsed `DATABASE_URL` hostname
- `DAILY_DIFF_ALLOW_NON_PRODUCTION=1` — local-dev opt-out; fail-closed when unset

For non-Neon Postgres (RDS, vanilla Postgres, etc.) `inet_server_addr()` may still be useful — the discriminator depends on the platform's networking topology. The principle holds (verify endpoint before any production-data interaction); the *signal* that carries discriminative power is platform-specific. Operators applying Pattern D to a new platform should empirically validate which signal actually varies between branches before relying on it.

### Pattern E: Verify DDL apply via structural existence check, not return-status alone

When a SQL apply tool reports "statement executed successfully" but executes only a subset of a multi-statement script, return-status is silent on the partial-apply. Verification against the reported return code produces a false-positive — operator believes the apply landed but production state hasn't changed.

**Symptom shape:** apply tool reports success, no errors logged, but EXISTS / COUNT queries against the expected post-apply schema (or data) return zero / not-found.

**Pre-flight pairing (operator-side, mechanical):**

```sql
-- After applying a CREATE TABLE migration:
SELECT EXISTS (
  SELECT 1 FROM information_schema.tables
  WHERE table_schema = 'sp' AND table_name = 'daily_diff_reports'
);
-- Must return true. False → apply did not land regardless of tool's success message.

-- For data migrations:
SELECT count(*) FROM sp.daily_diff_reports;
-- Compare against expected row count from the apply.
```

**Cost asymmetry:** structural check is 1-2 seconds; recovery from a silent partial-apply is hours of confused investigation when downstream behavior contradicts the "success" message.

Discovered during Phase 2 Track A Deliverable 2 migration apply on 2026-05-21. The migration's offline SQL (4 `CREATE TABLE` / `CREATE INDEX` / `ALTER TABLE` / `UPDATE sp.alembic_version` statements wrapped in `BEGIN; ... COMMIT;`) was pasted into Neon's web console SQL editor. Editor reported "Statement executed successfully" but only the final `COMMIT` statement actually executed — the `CREATE TABLE` statements were not applied. Verified post-apply via `SELECT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_schema='sp' AND table_name='daily_diff_reports')` → false. Recovery: chunked the DDL into 4 single-statement applies, each verified via EXISTS check post-apply. Cost: ~10 minutes once the partial-apply hypothesis was identified.

Two distinct lessons:
- **Tool-specific:** Neon's web console SQL editor does not reliably execute multi-statement blocks; chunk DDL into single statements when applying via the console. (Use `psql` for multi-statement when available; tool unreliability is the operator-side workaround.)
- **Pattern-general:** never trust return-status alone for DDL apply. Pair every apply with a structural verification query (EXISTS / COUNT / `\d` equivalent). Even when the tool is reliable, the structural check costs nothing and catches typos / silent failures in the DDL itself.

Pattern E complements Pattern D: D ensures the apply targets the right database; E ensures the apply actually landed. Both are mechanical, both are ~5-second checks, both have order-of-magnitude cost asymmetry vs. the silent-failure recovery.

### Pattern F: Verify whether a sport's records reach the matcher before committing to a bootstrap methodology

Before committing to a bootstrap scope-doc, verify whether the target sport's records are structurally H2H matchups (suitable for team/player bootstrap) or tournament-prop / outright-winner markets (suitable for scope-filter extension per Issue #160). The discrimination is gated on `INDIVIDUAL_SPORT_CODES` membership (`resolver/alias_tier/normalize.py:41-48`) plus the provider's market-structure taxonomy.

**Symptom shape:** operator commits to a bootstrap scope-doc (seed manifest + bootstrap script + tests) for a sport that turns out to have no per-fixture H2H records on the provider. Days of scope-doc + implementation work produce zero auto-apply lift because the records never reached the matcher in the first place — they were tournament-prop markets filtered at signal extraction (`resolver/kalshi.py:150-152` returns `None` for non-per_fixture records).

**Pre-flight discriminator (operator-side, ~30 minutes):**

1. Check whether the sport is in `INDIVIDUAL_SPORT_CODES` (personal-path → Pattern F applies) or team-path (standard H2H assumption, Pattern F still applies as confirmation).
2. Query production data for the target sport's market structure — series-ticker taxonomy (Kalshi) or `TOURNAMENT_NAME` / event structure (FL).
3. If tournament-prop-dominant or outright-winner-dominant: pivot to scope-filter extension (vocabulary addition to `KALSHI_PROP_MARKET_SEGMENTS`), not bootstrap.
4. If H2H-dominant: proceed with bootstrap methodology (KBL precedent for team-path; Tennis dedup precedent for personal-path).

**Cost asymmetry:** 30-minute code survey + 2-3 production queries prevents 5-7 days of wasted bootstrap scoping. Same epistemic shape as Pattern A (verify assumptions against production data before committing work), applied at the methodology-shape granularity.

Discovered during Day-22 (2026-05-22) Golf scope-filter pivot: the 5-sport zero-coverage cohort positioned Golf as the biggest single-sport bootstrap leverage (1,371 no_match/day). Pattern F diagnostic revealed Golf records on Kalshi are structurally tournament-prop-only — every series base (`KXPGATOUR`, `KXPGAR1LEAD`, `KXPGAMAKECUT`, etc.) attaches sub-markets to `(tournament_handle, year)` identity, never H2H fixtures. The Kalshi resolver returns `None` at signal extraction; Golf records never reach the matcher. Outcome: scope-filter extension (PR #181) instead of bootstrap. Days of wasted scoping prevented.

**Pattern F vs Pattern A:** Pattern A is content-stage discipline (verify the provider's actual generated forms before assuming alias completeness within a committed methodology). Pattern F is meta-stage discipline (verify the sport's record structure supports the methodology before committing to a scope-doc shape). Different epistemic layers — Pattern A asks "do my aliases match the data?", Pattern F asks "does the data reach the code path my aliases feed?"

### Pattern A.2: Discovery-before-commit at multiple granularities

**Sub-pattern of Pattern A.** The same epistemic discipline — "verify the assumed shape exists before committing work" — applies at every level of structural commitment, not just at query-result time:

| Granularity | What's verified | Example |
|---|---|---|
| **Seed content** (Pattern A original) | Provider's actual generated forms match the alias coverage in the manifest | KBL: "Goyang Skygunners" 2-token form caught by `sp.resolution_log` DISTINCT query |
| **Methodology shape** (Pattern F) | Sport's records reach the matcher at all | Golf: tournament-prop records never reach matcher; pivot to scope-filter |
| **Cohort prioritization** (Pattern G) | Daily volume signal reflects top-tier leverage, not long-tail saturation | Handball: 253/day signal was long-tail-dominated; 0/224 reachability against top-tier manifest |
| **Criterion design** (Tennis dedup) | Matcher-evidence signal carries real dupes, not just disambiguation noise | Tennis dedup: matcher candidate-set signal (`colliding_*_team_ids`) carries both real dupes AND search-space noise; structural feature extraction separates them |

The principle: before committing N days of work at any level, run a cheap verification (~30 minutes of queries / code survey) that the assumed shape exists. The cost-asymmetry is consistent across all levels — cheap pre-flight prevents expensive misdirection.

Discovered during Day-22 (2026-05-22) when two pivots in a single morning — Golf (methodology-shape granularity) and Handball (cohort-prioritization granularity) — demonstrated the same discipline at different levels. ~15 minutes of code survey caught what would have been days of wasted Golf bootstrap scoping; ~30 minutes of Pattern G queries caught the Handball cohort-framing error.

### Pattern G: Bootstrap-cohort 3-step diagnostic before committing to cohort prioritization

Daily-volume signals (records/day per sport from `sp.resolution_log` or daily-diff classifications) measure combined top-tier + long-tail population. Bootstrap value, however, is gated on top-tier reachability — the seed manifest scope is fundamentally capped by how many distinct teams can be curated. Long-tail-dominated sports may saturate the volume signal while remaining structurally unaddressable by bootstrap.

**Pre-flight discriminator (operator-side, ~30 minutes):**

1. **Daily volume** — sport's records/day in the unresolved (`no_match` + `signal_extraction_skipped`) population. The cohort-prioritization input.
2. **Top-tier vs long-tail split** — multi-day variant capture (Pattern A Q2-shape query over a 7-day window). Count distinct team-pairs; if the top-N pairs are dominated by `records_in_7d ≤ 2` (cron-duplicate rows of the same match, not multiple distinct matches), the sport is long-tail-dominated. Top-tier leagues run weekly cadence (~1-3 matches/club/week → ~2 records/7d per pair); long-tail leagues run daily cadence producing many pairs at 1 record each.
3. **Reachability of top-tier manifest** — draft a starter manifest of top-tier teams (20-50 entries, drawn from Q2-surfaced provider-forms per Pattern A, NOT general-knowledge canonicals), run Q6-shape reachability check against last-24h records. If `strict_tier_reachable_both ≥ 50%`, top-tier bootstrap delivers meaningful leverage. If `< 20%`, the bootstrap value is structurally low regardless of seed curation effort.

**Cost asymmetry:** 3-step diagnostic is ~30 minutes of operator-side queries; the alternative is 5-7 days of scope-doc + seed + bootstrap script + test work for a sport that won't deliver auto-apply lift.

**Empirical case studies (Day-22/25):**

| Sport | Step 1 (volume) | Step 2 (top-tier vs long-tail) | Step 3 (reachability) | Verdict |
|---|---|---|---|---|
| Golf | 1,371/day | N/A (Pattern F: tournament-prop, not H2H) | N/A | Scope-filter extension (PR #181), not bootstrap |
| Handball | 1,019 distinct/day (FL-only) | Long-tail dominant; top 11 pairs at records_in_7d=2 (cron duplicate); only 2/30 visible pairs were top-tier | 0/224 reachable against 47-team top-tier manifest | Dropped from bootstrap priorities |
| Rugby Union | 474 distinct/day (FL-only) | Long-tail dominant + possible FL sport-misclassification (Issue #183); Step 2 conclusive | — (Step 3 unnecessary) | Dropped |

**Generalization**: applies to all bootstrap-cohort prioritization, not just the Day-17 5-sport cohort. Whenever cohort selection uses a daily-volume signal as the leverage proxy, Pattern G's diagnostic discriminates "high daily volume" from "high top-tier bootstrap leverage." The two are not interchangeable (v1.5 amendment #11).
