# Tennis Cross-Format Dedup

Phase 2 workstream: merge cross-format duplicate player records in `sp.teams` where the same real-world Tennis player has multiple `team_id` rows due to name-form variations across providers.

Precedent: follows KBL bootstrap methodology (`docs/bootstraps/kbl-2025-26.md`) for operator-led scope + implementation discipline, but the underlying operation is merge-and-consolidate rather than seed-and-bootstrap.

---

## 1. Context and motivation

Tennis is the largest single-sport `review_queue` population (~4,613 pending records as of Day-21). Day-21 daily-diff measured Tennis matcher-capability at 24.1% — not 0% as the production cron's incremental rate implied (v1.5 amendment #8: matcher-capability vs incremental-apply distinction).

The gap between 24.1% and the team-path average (~70%) is partly driven by cross-format duplicates: the same player CAN exist under multiple `sp.teams` rows with different canonical forms (e.g., "Carlos Alcaraz" from FL vs "C. Alcaraz" from Kalshi's surname-only ticker). Most players have a single row; dedup targets the subset where duplication prevents strict-tier matching. When the resolver encounters a record with one form and the other form already has an alias-resolved `team_id`, the duplicate row prevents strict-tier matching from connecting the record to the right canonical.

Dedup collapses duplicate rows into a single canonical `team_id` per player. Post-dedup, both provider-forms alias-resolve to the same `team_id`, lifting collision-routed records from `review_queue` into strict-tier auto-apply.

### Why now

1. **Known scope.** Day-20 investigation estimated ~457 high-confidence + ~263 candidate-verification cross-format pairs. Bounded engineer-time (~3-5 days design + script + tests + verification).
2. **Empirical foundation.** Phase 2D.5 prerequisite: Tennis dedup must ship before FL-side alias coverage expansion (deferred per `PHASE_2D_DESIGN.md:33, 983`).
3. **Largest single-sport leverage.** 4,613 pending review_queue records; dedup lifts the collision-routed subset into auto-apply.
4. **Cohort bootstraps dropped.** Day-22 Pattern G diagnostic empirically rejected the 5-sport zero-coverage cohort framing (Golf: tournament-prop scope-filter; Handball + Rugby Union: long-tail dominated). Tennis dedup is the next-highest-leverage Phase 2 workstream.

### FL-only corroboration ceiling (v1.5 amendment #10)

Tennis has both Kalshi and FL records — it is NOT FL-only. Cross-provider corroboration IS available for Tennis. However, the dedup workstream specifically targets records where duplicate `team_id` rows prevent strict-tier matching. Post-dedup, records that previously collided on duplicate rows may reach strict tier (0.98 confidence, auto-apply) without needing corroboration. Records that still need alias/fuzzy tier benefit from corroboration when both providers carry the match.

## 2. Scope boundaries

### In scope

- **Identification** of cross-format duplicate pairs in `sp.teams WHERE sport_id = <tennis_sport_id>`. Two rows are duplicates when they represent the same real-world player but carry different `canonical_name` forms.
- **Merge operation** per pair: consolidate aliases, rewrite FK references, delete the duplicate row. Single transaction per pair (Phase 2D.3.1 discipline).
- **Two-phase batching**: Phase A (high-confidence pairs, automated merge) + Phase B (candidate-verification pairs, operator-approved merge).
- **Audit trail**: `sp.dedup_audit` table with pre-merge state snapshots per pair. Rollback from audit row.
- **Baseline-shift annotation** in `sp.baseline_shifts` at ship time — daily-diff Day-N+1 shows the expected Tennis matcher-capability lift.
- **Pattern A pre-flight discovery query** against production data before implementation locks. The 457/263 figures are Day-20 working estimates (no committed query derivation). Discovery query verifies: (a) cross-format duplicate structure exists in current production data, (b) matcher-evidence JSONB shape in `sp.resolution_log.reason_detail` carries the expected `home_team_id` / `away_team_id` keys, (c) concrete pair count meeting the refined F8 criterion (replaces working estimates with empirical numbers).

### Out of scope

- **Non-Tennis sports.** MMA/Boxing/Golf/Snooker/Darts have similar personal-path semantics but different population shapes. Tennis first; methodology transfers after validation.
- **FL-side alias coverage expansion** (Phase 2D.5). Dedup is a prerequisite; Phase 2D.5 ships after Tennis dedup validates the consolidated player population.
- **Fixture-level dedup.** If two duplicate players each have distinct fixtures, those fixtures may also be duplicates. Fixture dedup is a separate concern; this workstream merges the `sp.teams` layer only. Fixture dedup may surface as a follow-up if daily-diff shows unexpected fixture-count inflation post-merge.
- **review_queue re-resolution.** Post-merge, pending review_queue rows get JSONB-rewritten (F5: in-place candidate_fixtures team_id substitution), NOT re-resolved. The daily cron's re-resolution loop handles any records that can now strict-tier-match against the consolidated player.
- **`sp.resolution_log` rewriting.** Immutable audit trail; pre-merge decisions preserved for forensics.

## 3. Framing-question matrix

Operator decisions per question. Each row carries a default per precedent + rationale; operator confirms or overrides.

### F1 — Canonical-form selection at merge time

**Question:** When two `sp.teams` rows for the same player merge, which row's `canonical_name` wins the surviving row's slot?

**Decision:** Preserve the older `created_at` row's `canonical_name`; demote the newer row's canonical to alias on the survivor. Mirrors KBL F1 / PR #156 precedent (FL is §9.3 authoritative source; UPDATE-ing canonical creates drift).

Tiebreaker for same-day-created rows:
1. Prefer the row with more `sp.team_aliases` attached (more alias coverage = more provider-form evidence).
2. On tie: prefer the longer `canonical_name` form (e.g., "Carlos Alcaraz" over "C. Alcaraz" — the longer form is more informative for operator display).

### F2 — Alias preservation at merge time

**Question:** Both pre-merge rows have their own `sp.team_aliases` sets. Union them, dedupe by `(alias_normalized, source)`, or keep both rows' aliases as-is?

**Decision:** Union with dedupe — combine both alias sets. `INSERT ... ON CONFLICT (alias_normalized, source) DO NOTHING` handles the collision at write time. Preserves source attribution from both pre-merge histories.

### F3 — Source-value handling for pre-existing aliases

**Question:** Pre-merge rows may have aliases from `strict_tier`, `fuzzy_tier`, `manual_review`, `bootstrap_*` sources. What gets preserved?

**Decision:** Preserve all source values verbatim. No source rewriting during merge. Provenance fidelity matters for future audits tracing any alias back to its origin source.

### F4 — FK cascade design

**Question:** Which tables reference the to-be-deleted `team_id`, in what order?

**Answer (empirical, per schema survey):**

```
BEGIN; -- one pair per transaction (Phase 2D.3.1)

-- 0. Lock both rows at transaction start — fail-fast if either is already gone
--    (prevents concurrent-modification race if two merge operations target
--    overlapping team_ids, e.g., a 3-way duplicate A→B while C→B also runs).
SELECT id FROM sp.teams WHERE id IN (:canonical_id, :dupe_id) FOR UPDATE;
-- If either row missing, transaction aborts here (0 or 1 rows returned vs 2 expected).

-- 1. Copy dupe's aliases to canonical (dedupe via UNIQUE)
INSERT INTO sp.team_aliases (id, team_id, alias, alias_normalized, source, confidence, created_at)
SELECT gen_random_uuid(), :canonical_id, a.alias, a.alias_normalized,
       a.source, a.confidence, a.created_at
FROM sp.team_aliases a WHERE a.team_id = :dupe_id
ON CONFLICT (alias_normalized, source) DO NOTHING;

-- 2. Rewrite fixture home/away references (NO ACTION FK — must update before delete)
UPDATE sp.fixtures SET home_team_id = :canonical_id WHERE home_team_id = :dupe_id;
UPDATE sp.fixtures SET away_team_id = :canonical_id WHERE away_team_id = :dupe_id;

-- 3. Rewrite candidate_fixtures JSONB array entries (no FK enforcement)
UPDATE sp.review_queue
SET candidate_fixtures = (
  SELECT jsonb_agg(
    CASE WHEN elem #>> '{}' = :dupe_id_text THEN to_jsonb(:canonical_id_text) ELSE elem END
  ) FROM jsonb_array_elements(candidate_fixtures) elem
)
WHERE candidate_fixtures::text LIKE '%' || :dupe_id_text || '%';

-- 4. Pre-merge audit row (F7)
INSERT INTO sp.dedup_audit (canonical_id, merged_id, merged_at, pre_state)
VALUES (:canonical_id, :dupe_id, NOW(), :pre_state_jsonb);

-- 5. Delete dupe row (CASCADE on sp.team_aliases purges the originals; step 1's copies survive)
DELETE FROM sp.teams WHERE id = :dupe_id;

COMMIT;
```

Tables NOT rewritten:
- `sp.resolution_log` — immutable audit; pre-merge decisions stay for forensics
- `sp.fl_events` / `sp.kalshi_markets` / `sp.polymarket_markets` / `sp.oddsapi_events` — provider records reference fixtures via `fixture_id`, not `team_id` directly; fixture rows survive the rewrite at step 2

**Decision:** Concrete per schema survey. Step 0 (SELECT FOR UPDATE fail-fast) added per operator review to guard against concurrent-modification races. Cascade order confirmed.

### F5 — review_queue cleanup at merge time

**Question:** Pending review_queue rows pointing at to-be-merged team_ids: re-resolve (matcher re-runs), JSONB-rewrite in-place, or leave as orphans for re-resolution cron?

**Decision:** In-place JSONB rewrite (F4 step 3). Minimal disruption to operator's existing review context. Re-resolution risks producing different routing than the operator already saw; JSONB UPDATE preserves confidence / provider_title / created_at while pointing at the consolidated team_id.

Post-merge, the daily cron's re-resolution loop (§7.7) naturally picks up records that can now strict-tier-match against the consolidated player. No special drainage script needed.

### F6 — Batching strategy

**Question:** High-confidence (~457) vs candidate-verification (~263) — single PR or two-phase?

**Decision:** Two-phase shipping:

| Phase | Population | PR cadence | Operator effort | Gate |
|---|---|---|---|---|
| A | High-confidence (F8 criterion is the gate) | Single PR: scope-doc + migration + script + tests + apply | Per-PR review; `--dry-run` output reviewed before wet apply | F8 criterion met → automated merge, no per-pair admin UI |
| B | Candidate-verification (criterion-adjacent but needs eyes) | Separate PR after Phase A lands + daily-diff confirms expected delta | Operator approves each pair via admin review UI before merge | Per-pair admin UI approval |

Phase A is the bulk-leverage win. Phase B carries per-pair operator-judgment risk; slower cadence acceptable.

Both phases use per-pair transactions per Phase 2D.3.1. Phase A's PR includes `--dry-run` mode that produces a full merge report (which pairs, which aliases transfer, which fixtures rewrite) without applying. Operator reviews dry-run output before wet apply.

### F7 — Rollback plan

**Question:** If a merge surfaces unintended consequences, how does the operator revert?

**Decision:** Pre-merge snapshot per pair via `sp.dedup_audit` table (new, throw-away per scope doc §14). One row per merge with JSONB `pre_state` blob capturing:

- Both `sp.teams` rows (full column set)
- Both rows' full `sp.team_aliases` sets
- List of affected `sp.fixtures` IDs (home_team_id or away_team_id matched)
- List of affected `sp.review_queue` row IDs (candidate_fixtures contained the dupe team_id)

Rollback is a script that restores from the audit row. `pre_state` captures IDs (not full rows) for fixtures and review_queue to keep the JSONB size bounded; the actual row content is recoverable from the immutable tables.

Alternative considered: git-as-rollback-proxy via single-merge-per-commit discipline. Rejected — less structured, harder to automate partial rollback (e.g., revert pair #37 without reverting pairs #1-36).

### F8 — Success criterion

**Question:** What defines success — a target count (457 pairs merged) or a criterion match (all pairs meeting criterion X merged)?

**Decision:** Criterion-based, not target-count-based. The 457/263 figures are working estimates from Day-20 manual analysis (no committed query derivation). Success is "all pairs meeting the high-confidence criterion are merged" — the actual count is whatever the criterion query produces against current production data.

**High-confidence criterion (Phase A)** — all four conditions must hold:

1. **Same surname-anchor.** Both rows' `canonical_name` values, when processed through `resolver/alias_tier/normalize.py:structurally_normalize(sport_code='tennis')`, produce the same `surname` field.
2. **≥2 `sp.teams` rows** with `sport_id = <tennis_sport_id>` sharing the surname.
3. **Cross-format name-variant pattern.** One row's `canonical_name` is a format-variant of the other's: first-initial vs firstname (e.g., "C. Alcaraz" vs "Carlos Alcaraz"), token-rearrangement ("Alcaraz, Carlos" vs "Carlos Alcaraz"), or substring containment.
4. **Matcher-evidence (colliding_*_team_ids).** Both rows' `team_id`s appear together in the same `colliding_home_team_ids` or `colliding_away_team_ids` array within a single `sp.resolution_log` row, for ≥2 distinct `provider_record_id`s within the last 30 days. This is the safeguard against common-surname false-merges (Lee, Wang, Lopez, Kim, Garcia) — the resolver itself must have observed the collision by routing both `team_id`s as candidates for the same record.

**JSONB shape (validated Day-25 against production data):**

The collision signal is recorded atomically per `sp.resolution_log` row, not inferred across rows:

```json
{
  "sport": "Tennis",
  "is_personal": true,
  "home_collision": true,
  "colliding_home_team_ids": ["uuid-A", "uuid-B", ...],
  "colliding_away_team_ids": ["uuid-C", "uuid-D", ...],
  "home_provider_normalized": "alcaraz",
  "home_canonical": null
}
```

- `home_team_id` is `null` when the matcher couldn't pick one (collision case — ambiguous key per `resolver/aliases.py:115-119`).
- `colliding_home_team_ids` / `colliding_away_team_ids` carry the candidate UUID arrays that collided. Both team_ids appearing in the same array = the matcher observed the collision.
- `is_personal: true` gates personal-path sports (Tennis, MMA, Boxing, Golf, Snooker, Darts).

**Criterion query shape** (condition 4):

```sql
WITH tennis_collisions AS (
  SELECT
    provider_record_id,
    jsonb_array_elements_text(reason_detail->'colliding_home_team_ids')::uuid AS team_id
  FROM sp.resolution_log
  WHERE reason_detail->>'sport' = 'Tennis'
    AND reason_detail->'colliding_home_team_ids' IS NOT NULL
    AND jsonb_array_length(reason_detail->'colliding_home_team_ids') >= 2
    AND decided_at >= NOW() - INTERVAL '30 days'
  UNION ALL
  SELECT
    provider_record_id,
    jsonb_array_elements_text(reason_detail->'colliding_away_team_ids')::uuid AS team_id
  FROM sp.resolution_log
  WHERE reason_detail->>'sport' = 'Tennis'
    AND reason_detail->'colliding_away_team_ids' IS NOT NULL
    AND jsonb_array_length(reason_detail->'colliding_away_team_ids') >= 2
    AND decided_at >= NOW() - INTERVAL '30 days'
),
collision_pairs AS (
  SELECT
    LEAST(a.team_id, b.team_id) AS team_a,
    GREATEST(a.team_id, b.team_id) AS team_b,
    a.provider_record_id
  FROM tennis_collisions a
  JOIN tennis_collisions b
    ON a.provider_record_id = b.provider_record_id
    AND a.team_id < b.team_id
)
SELECT team_a, team_b,
       count(DISTINCT provider_record_id) AS shared_records
FROM collision_pairs
GROUP BY team_a, team_b
HAVING count(DISTINCT provider_record_id) >= 2
ORDER BY shared_records DESC;
```

No expensive self-join on the 2M-row resolution_log table — the collision signal is unpacked from JSONB arrays within rows, then paired. Runs in seconds on Neon serverless.

Conditions 1-3 are computable from `sp.teams` alone. Condition 4 uses the query above. The F8 criterion is the intersection of both result sets.

**Pattern A discovery query** (prerequisite before sections 4-7): the criterion query above, run against production, produces the concrete pair count meeting condition 4. Cross-reference against conditions 1-3 (surname-anchor + format-variant) to produce the final Phase A population. Replaces the 457/263 working estimates with empirical numbers.

**Candidate-verification criterion (Phase B):** pairs meeting conditions 1-3 but NOT condition 4 (no matcher-evidence within 30 days). These need operator eyes per pair — the name similarity is suggestive but the resolver hasn't independently observed the collision.
