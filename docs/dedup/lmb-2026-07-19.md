# LMB Duplicate-Canonical Dedup (Gate #3d)

Bounded data-reconciliation workstream: 14 LMB team pairs where two `sp.teams` rows point at the same real-world club under different canonical forms — one bare city/short name from the 2026-05-08 bootstrap, one full club name from `bootstrap_league_coverage` on 2026-05-28. Between 05-08 and 05-28 the bare rows accumulated ~325 fixtures each; after 05-28 the alias tier sees two candidates for every FL city-name payload, correctly refuses to guess, and routes to `review_queue`.

Precedent: follows `docs/dedup/tennis-2026-05-25.md` shape — same-shape scope-doc + snapshot-first execution — with a smaller, bounded target set and a fundamentally simpler operation (single UPDATE + alias migration + single DELETE per pair, no fixture re-pointing).

**Do NOT execute until the pre-flight discovery in §1 reproduces exactly 14 rows against production.** If the row count drifts, either a new duplicate emerged, the operator's earlier count changed, or the predicate has drifted from the guard — investigate before writing.

**Filename vs execution date** (recorded Day-51): this file's name and the snapshot table `sp.lmb_dedup_snapshot_2026_07_19` both carry `2026-07-19` from pre-execution drafting. The batch actually executed on **2026-07-16** (`SELECT NOW()` at session start). The authoritative temporal record is `sp.baseline_shifts.event_date = 2026-07-16` on row `81b9cc23` (Day-51 annotation); the filename and snapshot-table name are kept as document/table identifiers that happen to differ from the execution date. Daily-diff reads `event_date`, not the filename, so the drift is inert for measurement purposes — but a future reader glancing at `ls docs/dedup/` should not read `_07_19` as "the merge ran on the 19th."

## 1. Context and motivation

### Root cause with a timestamp

- **2026-05-08 15:33:16 bootstrap** (identity of the script is a Day-49+ audit item; not blocking) created 14 bare LMB rows with city / short-form names ("Campeche", "Oaxaca", "Jalisco", "Chihuahua", "Diablos Rojos", etc.). `country_code = NULL`.

After 05-28 two canonicals compete for every FL city payload, the matcher correctly refuses (`home_collision: true`, `away_team_id: null`), and the class has failed every night since. **The coverage bootstrap caused the regression; the matcher is behaving correctly.**

### Controlled natural experiment (Day-49)

Auto-applied via the alias tier's `token_set` scorer against the full canonical `Conspiradores de Querétaro` (Day-48 F8 break watched this exact path fire with `token_set_contribution: 0.3`).
- **Campeche**: bare duplicate exists (`ac8c7ffd "Campeche"` + `a941cb3f "Piratas de Campeche"`). `away_collision: true`, `away_team_id: null` → `review_queue` every pass.

Same league, same city → club shape, same alias tier, one variable: whether a bare duplicate exists. Verified on real `sp.resolution_log` rows from the last 48h (records `l2WSNvRa`, `d2jpmefB`, `GO6YePxq` — all `reason_code = 'review_queue'`, `resolver_version = alias@2c.0`, strict tier logging `fail_reason = alias_resolution_incomplete` with `home_resolved: true` / `away_resolved: false`).

**The alias tier's token-set scorer bridges city → club fine when unopposed.** It only fails when two canonicals compete. Same token-subset property that defeated F8 Attempt 3 — here it's the mechanism that will fix the class once the second candidate disappears.

### The `country_code` discriminator

The 05-08 bare rows carry `country_code = NULL`; the 05-28 `bootstrap_league_coverage` rows carry `country_code = 'MEX'`. This is what separates the real 13-pair target set from false positives like `Chihuahua` → `El Paso Chihuahuas` (both created 2026-05-08, both `country_code = NULL`, different real-world clubs). Discovery in §1 surfaces `country_code` as a confirmation column, not a hard filter — operator eyeballs the shortlist.

### Why this is a bounded workstream

- **14 pairs, verified** (13 confirmed Day-49 + Chihuahua/Dorados de Chihuahua confirmed Day-49 with same 05-08 / 05-28 / NULL / 'MEX' signature).
- **~325 fixtures per bare row, ZERO fixtures per full row** — no parallel fixture set to reconcile, no duplicate-fixture hazard on merge.
- **No fixture re-pointing needed** under direction (b) below.
- Baseline shift is meaningful (~370 records/week freed from `review_queue`) but the scope is finite and known.

## 3. Direction — (b) with bootstrap-idempotency verified

Two directions were considered:

- **(a)** Re-point ~325 fixtures bare → full, migrate aliases, delete bare.
- **(b)** Rename the bare row's `canonical_name` / `normalized_name` to the full club form, migrate any full-row aliases to the bare row, delete the (unreferenced, zero-fixture) full row.

**Direction (b) chosen.** Not primarily because it's cheaper — the deciding argument is `sp.resolution_log` history preservation:

> The bare row's `team_id` is referenced by every historical `resolution_log` decision. Direction (a) orphans that history and would need a `reason_detail` sweep to keep decisions reproducible. Direction (b) preserves it — the row keeps its id, only its identity string upgrades.

### Bootstrap-idempotency check (`scripts/bootstrap_lmb.py`)

`bootstrap_lmb.py:156-163`:

```python
normalized = normalize_name(canonical_name)  # from LMB_TEAMS_SEED
existing = existing_teams_by_normalized.get(normalized)
if existing is None:
    # insert new team
else:
    # reuse existing_id, backfill country_code if needed
```

The existence lookup keys on `normalized_name` derived from the seed's FULL canonical (`"Piratas de Campeche"` → `"piratas de campeche"`). Post-merge, the renamed bare row's `normalized_name` IS `"piratas de campeche"` → the next `bootstrap_lmb.py` run finds it, reuses its team_id, does NOT insert. **Bootstrap is idempotent-safe under direction (b) precisely because it keys on the correct-canonical shape we're renaming into.**

Alias step (`bootstrap_lmb.py:218-280`) uses `NOT EXISTS` on `(team_id, alias_normalized)` per-team AND `(alias_normalized, source)` globally. Post-merge the seed's aliases (`"Campeche"`, `"Piratas"`, `"Piratas de Campeche"` for the Campeche pair) all exist on the renamed bare row → skipped. If any weren't migrated, the global-conflict check keeps them from landing on the wrong team. **Bootstrap is a safe no-op on subsequent runs.**

### Deciding arguments recap

| | (a) Re-point fixtures | (b) Rename + migrate + delete |
|---|---|---|
| Fixture rows touched | ~325 | 0 |
| `sp.resolution_log` history preservation | Orphaned (needs `reason_detail` sweep) | Preserved (same team_id) |
| Alias migration required | bare → full | full → bare (usually smaller set) |
| Bootstrap re-run safety | ambiguous | verified safe |
| Reversibility cost | 325 fixture re-points + alias revert | 1 UPDATE revert + 1 INSERT revert + 1 DELETE revert |

## 5. §1 — Discovery + confirmation

Reproduces the Day-49 finding. Filters ONLY on the substring/token relationship within Baseball — provenance (created_at, country_code) is surfaced as COLUMNS, not baked into the WHERE clause. This is deliberate: hard-filtering the query on the same signature that defines the 14 known targets makes the query incapable of failing honestly (it would return exactly 14 by construction, hiding drift). The query below CAN return other rows (a new duplicate with different provenance, El Paso Chihuahuas, the KBO/NPB pairs, the Twins) — the operator eyeballs the shortlist against the expected signature.

```sql
WITH baseball_teams AS (
    SELECT
        t.id,
        t.canonical_name,
        t.normalized_name,
        t.country_code,
        t.created_at,
        s.name AS sport
    FROM sp.teams t
    JOIN sp.sports s ON s.id = t.sport_id
    WHERE s.name = 'Baseball'
),
substring_pairs AS (
    -- Pair when the shorter row's normalized_name is a whole-word
    -- substring of the longer row's normalized_name. Token-subset
    -- semantics would be tighter but require Python (see the F8
    -- §2c(d) reasoning in docs/reresolution/f8-procedure.md); for
    -- discovery, whole-word substring surfaces the same shortlist
    -- plus a small number of false-positive candidates the operator
    -- eyeballs. NO provenance filter here — we WANT surprises to
    -- show up.
    SELECT
        bare.id                AS bare_team_id,
        bare.canonical_name    AS bare_canonical,
        bare.normalized_name   AS bare_normalized,
        bare.country_code      AS bare_country_code,
        bare.created_at        AS bare_created_at,
        full_row.id                 AS full_team_id,
        full_row.canonical_name     AS full_canonical,
        full_row.normalized_name    AS full_normalized,
        full_row.country_code       AS full_country_code,
        full_row.created_at         AS full_created_at
    FROM baseball_teams bare
    JOIN baseball_teams full_row
      ON bare.id != full_row.id
     -- Bare is the SHORTER row; substring test protects against
     -- pairing (A, A||B) both directions.
     AND length(full_row.normalized_name) > length(bare.normalized_name)
     AND full_row.normalized_name LIKE '%' || bare.normalized_name || '%'
)
SELECT
    p.bare_team_id,
    p.bare_canonical,
    p.bare_country_code,
    p.bare_created_at::date AS bare_created_date,
    p.full_team_id,
    p.full_canonical,
    p.full_country_code,
    p.full_created_at::date AS full_created_date,
    -- Signature match — the expected 14 targets carry:
    --   bare.created_at::date = '2026-05-08'
    --   full.created_at::date = '2026-05-28'
    --   bare.country_code IS NULL
    --   full.country_code = 'MEX'
    -- Anything without this signature is NOT in the target set for
    -- this workstream — El Paso Chihuahuas / KBO / NPB / Twins all
    -- fall outside it. Operator eyeballs; do NOT let non-signature
    -- rows into §3.
    (
        p.bare_created_at::date = DATE '2026-05-08'
        AND p.full_created_at::date = DATE '2026-05-28'
        AND p.bare_country_code IS NULL
        AND p.full_country_code = 'MEX'
    ) AS matches_lmb_dedup_signature,
    -- Fixture counts — expect ~21-30 on bare, 0 on full for signature-matches:
    (SELECT count(*) FROM sp.fixtures fx
     WHERE fx.home_team_id = p.bare_team_id OR fx.away_team_id = p.bare_team_id) AS bare_fixture_count,
    (SELECT count(*) FROM sp.fixtures fx
     WHERE fx.home_team_id = p.full_team_id OR fx.away_team_id = p.full_team_id) AS full_fixture_count,
    -- Alias counts — for planning the migration step:
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = p.bare_team_id) AS bare_alias_count,
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = p.full_team_id) AS full_alias_count
FROM substring_pairs p
ORDER BY matches_lmb_dedup_signature DESC, p.bare_canonical;
```

**Expected shape of the result** — the query is designed to surprise, but not confuse:

- **>= 14 rows total.** Non-signature rows land at the bottom (ORDER BY signature-first).
- **Exactly 14 rows with `matches_lmb_dedup_signature = true`** — the target set. Each has `bare_fixture_count` between 21 and 30 and `full_fixture_count = 0`. This is the operator's authoritative dedup batch.
- **Remaining rows have `matches_lmb_dedup_signature = false`** — the known eyeball list:
  - `El Paso Chihuahuas` paired with `Dorados de Chihuahua` (both 05-08, both country_code NULL) — different real-world clubs, do not merge.
  - `Twins` (Minnesota MLB) x `LG Twins` (KBO) — cross-competition, Gate #3c.
  - Four KBO/NPB substring pairs — both created 05-08, different provenance, separate investigation.

**Non-signature rows that AREN'T on the known eyeball list**: a new duplicate emerged since Day-49 with different provenance. STOP; investigate before running §2 / §3 on any pair. Do not silently expand the batch.

**If signature-match count is NOT 14**: STOP. Either a signature-matching duplicate was created since Day-49 (add to batch after eyeballing), one of the 14 was manually reconciled between Day-49 and now (remove from batch and verify byte-exact reconciliation), or the query has a bug. Investigate before writes.

**If `full_fixture_count > 0` on any signature-match row**: STOP. Direction (b) assumes zero fixtures on the full row. Non-zero means fixtures accumulated somewhere unexpected; do NOT delete the full row without a fixture re-point plan (revert to direction (a) for that pair only).

## 6. §2 — Snapshot capture per pair

For each of the 14 pairs, capture the pre-merge state to a snapshot table so restore is byte-exact reversal. Create the snapshot table once:

```sql
CREATE TABLE IF NOT EXISTS sp.lmb_dedup_snapshot_2026_07_19 (
    pair_id                    text        NOT NULL,   -- e.g. 'campeche'
    captured_at                timestamptz NOT NULL DEFAULT NOW(),
    bare_team_id               uuid        NOT NULL,
    bare_canonical_pre         text        NOT NULL,
    bare_normalized_pre        text        NOT NULL,
    bare_country_code_pre      text,
    -- full_created_at_pre and full_sport_id are LOAD-BEARING for §7
    -- Step 1: recreating the full row with its original created_at
    -- and sport_id is what makes restore byte-exact. Reading
    -- created_at off the bare row (same session's snapshot time) or
    -- off the first alias (which may not exist, and is a different
    -- column on a different table anyway) would silently drift the
    -- restored row's identity. Required NOT NULL — capture-time
    -- SELECT below cannot skip them.
    full_team_id               uuid        NOT NULL,
    full_canonical_pre         text        NOT NULL,
    full_normalized_pre        text        NOT NULL,
    full_country_code_pre      text,
    full_created_at_pre        timestamptz NOT NULL,
    full_sport_id              int         NOT NULL,
    -- Full alias set on both rows, captured verbatim for restore:
    bare_aliases_pre           jsonb       NOT NULL,
    full_aliases_pre           jsonb       NOT NULL
);
```

Per pair, insert one snapshot row. The `full_created_at_pre` and `full_sport_id` values come from `SELECT ... FROM sp.teams WHERE id = '{FULL_TEAM_ID}'`, NOT from any derived source — reading them off the bare row or the alias set would silently break restore:

```sql
INSERT INTO sp.lmb_dedup_snapshot_2026_07_19 (
    pair_id,
    bare_team_id, bare_canonical_pre, bare_normalized_pre, bare_country_code_pre,
    full_team_id, full_canonical_pre, full_normalized_pre, full_country_code_pre,
    full_created_at_pre, full_sport_id,
    bare_aliases_pre, full_aliases_pre
)
SELECT
    '{PAIR_ID}',
    '{BARE_TEAM_ID}'::uuid,
    (SELECT canonical_name  FROM sp.teams WHERE id = '{BARE_TEAM_ID}'::uuid),
    (SELECT normalized_name FROM sp.teams WHERE id = '{BARE_TEAM_ID}'::uuid),
    (SELECT country_code    FROM sp.teams WHERE id = '{BARE_TEAM_ID}'::uuid),
    '{FULL_TEAM_ID}'::uuid,
    (SELECT canonical_name  FROM sp.teams WHERE id = '{FULL_TEAM_ID}'::uuid),
    (SELECT normalized_name FROM sp.teams WHERE id = '{FULL_TEAM_ID}'::uuid),
    (SELECT country_code    FROM sp.teams WHERE id = '{FULL_TEAM_ID}'::uuid),
    (SELECT created_at      FROM sp.teams WHERE id = '{FULL_TEAM_ID}'::uuid),
    (SELECT sport_id        FROM sp.teams WHERE id = '{FULL_TEAM_ID}'::uuid),
    COALESCE(
        (SELECT jsonb_agg(row_to_json(ta.*))
         FROM sp.team_aliases ta WHERE ta.team_id = '{BARE_TEAM_ID}'::uuid),
        '[]'::jsonb
    ),
    COALESCE(
        (SELECT jsonb_agg(row_to_json(ta.*))
         FROM sp.team_aliases ta WHERE ta.team_id = '{FULL_TEAM_ID}'::uuid),
```

## 7. §3 — Merge sequence per pair (5 steps, ONE transaction per pair)

**Blast-radius discipline**: 14 transactions, NOT one super-transaction. A failure in pair 7 must not block pairs 1-6 from committing or pairs 8-14 from starting cleanly.

**Why re-point, not copy** (Day-50 correction after Campeche pilot):

The earlier draft of Step 2 was `INSERT INTO sp.team_aliases (id, team_id, ...) SELECT gen_random_uuid(), '{BARE_TEAM_ID}', ... FROM sp.team_aliases WHERE team_id = '{FULL_TEAM_ID}' AND NOT EXISTS (...)`. That violates the global `uq_team_aliases_alias_normalized_source UNIQUE (alias_normalized, source)` constraint the instant the INSERT fires — the full row still holds the same `(alias_normalized, source)` pair, and the new row would duplicate it. Campeche's first transaction aborted at Step 2 (correctly, atomically, nothing half-written).

Re-point via `UPDATE sp.team_aliases SET team_id = '{BARE_TEAM_ID}' WHERE team_id = '{FULL_TEAM_ID}' AND ...` sidesteps the constraint entirely — an UPDATE that changes `team_id` never touches `(alias_normalized, source)`, so the unique index is never challenged. Rows keep their original `id` and `created_at`, which is better restore fidelity than minting fresh UUIDs would have been. `AND alias_normalized NOT IN (SELECT alias_normalized FROM sp.team_aliases WHERE team_id = '{BARE_TEAM_ID}')` skips aliases the bare row already has; the follow-up DELETE drops those.

Per pair, in this exact order (order matters — the global `uq_team_aliases_alias_normalized_source` constraint and the FK from `sp.team_aliases.team_id` to `sp.teams.id` both require the sequence below):

```sql
BEGIN;

-- Step 1: RENAME bare row's canonical + normalized to the full club form.
--
-- LOAD-BEARING: read `{FULL_NORMALIZED}` from the snapshot's
-- full_normalized_pre column, NEVER derive it locally by re-running
-- normalize_name(canonical) or by hand. Three of the 14 targets carry
-- diacritics ("Diablos Rojos del México", "El Águila de Veracruz",
-- "Leones de Yucatán") whose stored normalized forms are unaccented
-- ("diablos rojos del mexico", "el aguila de veracruz",
-- "leones de yucatan") because the 05-28 bootstrap ran the payload
-- through `resolver._normalize.normalize_name` (NFD + combining-mark
-- strip). If Step 1 writes any other normalized form — for example,
-- one derived from a locally-typed canonical without accent-strip —
-- `bootstrap_lmb.py:156-163`'s next run does
-- `existing_teams_by_normalized.get(normalize_name(canonical))` and
-- FAILS to find the renamed row, then INSERTs the duplicate back
-- silently. The rename would "succeed" and the duplicate class
-- would return on the next bootstrap.
--
-- Snapshot's full_normalized_pre column carries the exact string
-- the 05-28 bootstrap wrote; substitute it verbatim.
UPDATE sp.teams
SET canonical_name  = '{FULL_CANONICAL}',
    normalized_name = '{FULL_NORMALIZED}',
    country_code    = 'MEX'
WHERE id = '{BARE_TEAM_ID}';
-- Expected: 1 row updated.

-- Step 2: RE-POINT full row's aliases → bare row's team_id, but only
-- those whose alias_normalized doesn't already exist on the bare row.
-- UPDATE preserves the original alias id AND created_at, which
-- matters for restore fidelity in §7 and matches Campeche pilot's
-- empirical behavior (bare's original alias id 8c522d43 and full's
-- re-pointed ids 4714ef1b / f0b51c67 all preserved).
UPDATE sp.team_aliases
SET team_id = '{BARE_TEAM_ID}'
WHERE team_id = '{FULL_TEAM_ID}'
  AND alias_normalized NOT IN (
      SELECT alias_normalized
      FROM sp.team_aliases
      WHERE team_id = '{BARE_TEAM_ID}'
  );
-- Expected: rows re-pointed == full_alias_count − overlap. For
-- Campeche this was 2 (piratas + piratas de campeche re-pointed;
-- redundant campeche/bootstrap_league_coverage skipped).

-- Step 3: DELETE the aliases still attached to the full row — these
-- are the ones Step 2's NOT IN filter skipped (their alias_normalized
-- already exists on the bare row under a different source). For
-- Campeche this was 1 row (campeche/bootstrap_league_coverage, id
-- 4714ef1b, redundant with campeche/legacy_bootstrap on the bare
-- row).
DELETE FROM sp.team_aliases WHERE team_id = '{FULL_TEAM_ID}';
-- Expected: rows deleted == overlap count (full_alias_count minus
-- rows re-pointed in Step 2). May be 0.

-- Step 4: Reference-scan safety check. If the full row is referenced
-- by anything other than sp.team_aliases (already handled), this
-- SELECT surfaces it before step 5 tries to delete. Do NOT proceed
-- to step 5 if this returns non-zero.
SELECT 'fixtures_home' AS ref, count(*) AS n FROM sp.fixtures WHERE home_team_id = '{FULL_TEAM_ID}'
UNION ALL SELECT 'fixtures_away', count(*)     FROM sp.fixtures WHERE away_team_id = '{FULL_TEAM_ID}';
```

## 11. §7 — Rollback per pair (snapshot-driven, matches §3's re-point semantics)

If verification fails on any pair, or a later diagnostic surfaces a bug in the merge sequence, restore is snapshot-driven and byte-exact.

**Rollback logic mirrors §3's re-point semantics** (Day-50 correction after Campeche pilot). Per full-row alias captured in `full_aliases_pre`, the current state of the alias's original `id` distinguishes two cases:

1. **Alias id STILL EXISTS in `sp.team_aliases`** — it was re-pointed to the bare row in §3 Step 2. Restore by `UPDATE ... SET team_id = full` back. Do NOT re-INSERT (would collide on `pk_team_aliases`).
2. **Alias id NO LONGER EXISTS in `sp.team_aliases`** — it was hit by §3 Step 3's DELETE because its `alias_normalized` overlapped with a bare-row alias. Restore by `INSERT ... SELECT` from `full_aliases_pre` with the original id.

The snapshot has everything needed: original ids AND created_at captured verbatim in §6. `NOT EXISTS (SELECT 1 FROM sp.team_aliases WHERE id = <alias.id>)` distinguishes the two cases at rollback time — no need to re-derive overlap from `bare_aliases_pre` (that reasoning is what §3's UPDATE... NOT IN already handled at merge time).

Per pair:

```sql
BEGIN;

-- Step 1: Restore the full row with its original team_id, sport_id,
-- and created_at — all captured pre-merge in §6's snapshot.
INSERT INTO sp.teams (id, sport_id, canonical_name, normalized_name, country_code, created_at)
SELECT s.full_team_id,
       s.full_sport_id,
       s.full_canonical_pre,
       s.full_normalized_pre,
       s.full_country_code_pre,
       s.full_created_at_pre
FROM sp.lmb_dedup_snapshot_2026_07_19 s
WHERE s.pair_id = '{PAIR_ID}';
-- Expected: 1 row inserted. Runs first because Step 2's UPDATE
-- needs the full row as a valid FK target.

-- Step 2: Re-point aliases whose original ids STILL EXIST — these
-- were re-pointed to the bare row by §3 Step 2. Reverse by UPDATE
-- back to the full row. UPDATE doesn't touch (alias_normalized,
-- source), so the global unique index isn't challenged — same
-- reasoning as §3's re-point.
UPDATE sp.team_aliases
SET team_id = (SELECT full_team_id FROM sp.lmb_dedup_snapshot_2026_07_19
                WHERE pair_id = '{PAIR_ID}')
WHERE id IN (
    SELECT (a->>'id')::uuid
    FROM sp.lmb_dedup_snapshot_2026_07_19 s,
         jsonb_array_elements(s.full_aliases_pre) a
    WHERE s.pair_id = '{PAIR_ID}'
      AND EXISTS (
          SELECT 1 FROM sp.team_aliases ta
          WHERE ta.id = (a->>'id')::uuid
      )
);
-- Expected: rows updated == count of full_aliases_pre entries whose
-- id still exists in sp.team_aliases (i.e., the re-pointed subset).

-- Step 3: Re-insert aliases whose original ids NO LONGER EXIST —
-- these were dropped by §3 Step 3's DELETE. Re-INSERT from
-- snapshot with the original id AND created_at preserved.
INSERT INTO sp.team_aliases (id, team_id, alias, alias_normalized, source, confidence, created_at)
SELECT (a->>'id')::uuid,
       s.full_team_id,
       a->>'alias',
       a->>'alias_normalized',
       a->>'source',
       (a->>'confidence')::numeric,
       (a->>'created_at')::timestamptz
FROM sp.lmb_dedup_snapshot_2026_07_19 s,
     jsonb_array_elements(s.full_aliases_pre) a
WHERE s.pair_id = '{PAIR_ID}'
  AND NOT EXISTS (
      SELECT 1 FROM sp.team_aliases ta
      WHERE ta.id = (a->>'id')::uuid
  );
-- Expected: rows inserted == count of full_aliases_pre entries whose
-- id does NOT currently exist (i.e., the overlap-dropped subset).
-- Step 2's re-pointed count + Step 3's re-inserted count ==
-- jsonb_array_length(full_aliases_pre).

-- Step 4: Revert bare row's canonical / normalized / country_code.
UPDATE sp.teams
SET canonical_name  = (SELECT bare_canonical_pre    FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}'),
    normalized_name = (SELECT bare_normalized_pre   FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}'),
    country_code    = (SELECT bare_country_code_pre FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}')
WHERE id = (SELECT bare_team_id FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}');
-- Expected: 1 row updated.

-- Step 5: Verify byte-exact restore (INSIDE the transaction; ROLLBACK
-- if any assertion fails).
SELECT
    (SELECT canonical_name  FROM sp.teams WHERE id = s.bare_team_id) = s.bare_canonical_pre  AS bare_canonical_restored,
    (SELECT normalized_name FROM sp.teams WHERE id = s.bare_team_id) = s.bare_normalized_pre AS bare_normalized_restored,
    (SELECT count(*) FROM sp.teams        WHERE id = s.full_team_id) = 1                       AS full_row_restored,
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = s.full_team_id)
        = jsonb_array_length(s.full_aliases_pre)                                                AS full_aliases_restored,
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = s.bare_team_id)
        = jsonb_array_length(s.bare_aliases_pre)                                                AS bare_aliases_restored
FROM sp.lmb_dedup_snapshot_2026_07_19 s
WHERE s.pair_id = '{PAIR_ID}';
-- Expected: all five columns TRUE. If any FALSE, ROLLBACK.

COMMIT;
```
