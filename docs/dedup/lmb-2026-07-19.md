# LMB Duplicate-Canonical Dedup (Gate #3d)

Bounded data-reconciliation workstream: 14 LMB team pairs where two `sp.teams` rows point at the same real-world club under different canonical forms — one bare city/short name from the 2026-05-08 bootstrap, one full club name from `bootstrap_league_coverage` on 2026-05-28. Between 05-08 and 05-28 the bare rows accumulated ~325 fixtures each; after 05-28 the alias tier sees two candidates for every FL city-name payload, correctly refuses to guess, and routes to `review_queue`.

Precedent: follows `docs/dedup/tennis-2026-05-25.md` shape — same-shape scope-doc + snapshot-first execution — with a smaller, bounded target set and a fundamentally simpler operation (single UPDATE + alias migration + single DELETE per pair, no fixture re-pointing).

**Do NOT execute until the pre-flight discovery in §1 reproduces exactly 14 rows against production.** If the row count drifts, either a new duplicate emerged, the operator's earlier count changed, or the predicate has drifted from the guard — investigate before writing.

---

## 1. Context and motivation

### Root cause with a timestamp

- **2026-05-08 15:33:16 bootstrap** (identity of the script is a Day-49+ audit item; not blocking) created 14 bare LMB rows with city / short-form names ("Campeche", "Oaxaca", "Jalisco", "Chihuahua", "Diablos Rojos", etc.). `country_code = NULL`.
- **2026-05-28 19:41:10 `bootstrap_league_coverage`** (`scripts/bootstrap_lmb.py`) created the same 14 clubs under proper canonical names ("Piratas de Campeche", "Guerreros de Oaxaca", "Charros de Jalisco", "Dorados de Chihuahua", "Diablos Rojos del México", etc.). `country_code = 'MEX'`.

Between 05-08 and 05-28 the bare rows were the only match for FL city-name payloads, so they accumulated fixtures. After 05-28 two canonicals compete for every FL city payload, the matcher correctly refuses (`home_collision: true`, `away_team_id: null`), and the class has failed every night since. **The coverage bootstrap caused the regression; the matcher is behaving correctly.**

### Controlled natural experiment (Day-49)

- **Querétaro**: NO bare duplicate exists. Resolves cleanly nightly — `home_collision: false`, `home_ratio: 1.0`, `home_team_id = 2cb54cf9`, `reason_code = 'alias'`, `resolver_version = alias@2c.0`. Auto-applied via the alias tier's `token_set` scorer against the full canonical `Conspiradores de Querétaro` (Day-48 F8 break watched this exact path fire with `token_set_contribution: 0.3`).
- **Campeche**: bare duplicate exists (`ac8c7ffd "Campeche"` + `a941cb3f "Piratas de Campeche"`). `away_collision: true`, `away_team_id: null` → `review_queue` every pass.

Same league, same city → club shape, same alias tier, one variable: whether a bare duplicate exists. Verified on real `sp.resolution_log` rows from the last 48h (records `l2WSNvRa`, `d2jpmefB`, `GO6YePxq` — all `reason_code = 'review_queue'`, `resolver_version = alias@2c.0`, strict tier logging `fail_reason = alias_resolution_incomplete` with `home_resolved: true` / `away_resolved: false`).

**The alias tier's token-set scorer bridges city → club fine when unopposed.** It only fails when two canonicals compete. Same token-subset property that defeated F8 Attempt 3 — here it's the mechanism that will fix the class once the second candidate disappears.

### The `country_code` discriminator

The 05-08 bare rows carry `country_code = NULL`; the 05-28 `bootstrap_league_coverage` rows carry `country_code = 'MEX'`. This is what separates the real 13-pair target set from false positives like `Chihuahua` → `El Paso Chihuahuas` (both created 2026-05-08, both `country_code = NULL`, different real-world clubs). Discovery in §1 surfaces `country_code` as a confirmation column, not a hard filter — operator eyeballs the shortlist.

### Why this is a bounded workstream

- **14 pairs, verified** (13 confirmed Day-49 + Chihuahua/Dorados de Chihuahua confirmed Day-49 with same 05-08 / 05-28 / NULL / 'MEX' signature).
- **~325 fixtures per bare row, ZERO fixtures per full row** — no parallel fixture set to reconcile, no duplicate-fixture hazard on merge.
- **No fixture re-pointing needed** under direction (b) below.
- Baseline shift is meaningful (~370 records/week freed from `review_queue`) but the scope is finite and known.

## 3. Direction — (b) with bootstrap-idempotency verified

Two directions were considered:

- **(a)** Re-point ~325 fixtures bare → full, migrate aliases, delete bare.
- **(b)** Rename the bare row's `canonical_name` / `normalized_name` to the full club form, migrate any full-row aliases to the bare row, delete the (unreferenced, zero-fixture) full row.

**Direction (b) chosen.** Not primarily because it's cheaper — the deciding argument is `sp.resolution_log` history preservation:

> The bare row's `team_id` is referenced by every historical `resolution_log` decision. Direction (a) orphans that history and would need a `reason_detail` sweep to keep decisions reproducible. Direction (b) preserves it — the row keeps its id, only its identity string upgrades.

### Bootstrap-idempotency check (`scripts/bootstrap_lmb.py`)

`bootstrap_lmb.py:156-163`:

```python
normalized = normalize_name(canonical_name)  # from LMB_TEAMS_SEED
existing = existing_teams_by_normalized.get(normalized)
if existing is None:
    # insert new team
else:
    # reuse existing_id, backfill country_code if needed
```

The existence lookup keys on `normalized_name` derived from the seed's FULL canonical (`"Piratas de Campeche"` → `"piratas de campeche"`). Post-merge, the renamed bare row's `normalized_name` IS `"piratas de campeche"` → the next `bootstrap_lmb.py` run finds it, reuses its team_id, does NOT insert. **Bootstrap is idempotent-safe under direction (b) precisely because it keys on the correct-canonical shape we're renaming into.**

Alias step (`bootstrap_lmb.py:218-280`) uses `NOT EXISTS` on `(team_id, alias_normalized)` per-team AND `(alias_normalized, source)` globally. Post-merge the seed's aliases (`"Campeche"`, `"Piratas"`, `"Piratas de Campeche"` for the Campeche pair) all exist on the renamed bare row → skipped. If any weren't migrated, the global-conflict check keeps them from landing on the wrong team. **Bootstrap is a safe no-op on subsequent runs.**

### Deciding arguments recap

| | (a) Re-point fixtures | (b) Rename + migrate + delete |
|---|---|---|
| Fixture rows touched | ~325 | 0 |
| `sp.resolution_log` history preservation | Orphaned (needs `reason_detail` sweep) | Preserved (same team_id) |
| Alias migration required | bare → full | full → bare (usually smaller set) |
| Bootstrap re-run safety | ambiguous | verified safe |
| Reversibility cost | 325 fixture re-points + alias revert | 1 UPDATE revert + 1 INSERT revert + 1 DELETE revert |

## 5. §1 — Discovery + confirmation

Reproduces the Day-49 finding. Must return exactly 14 rows for `sport = 'Baseball'` matching the LMB signature before any writes begin.

```sql
WITH baseball_teams AS (
    SELECT
        t.id,
        t.canonical_name,
        t.normalized_name,
        t.country_code,
        t.created_at,
        s.name AS sport
    FROM sp.teams t
    JOIN sp.sports s ON s.id = t.sport_id
    WHERE s.name = 'Baseball'
),
substring_pairs AS (
    -- Bare candidates: 05-08 provenance, country_code NULL.
    -- Full candidates: 05-28 provenance, country_code 'MEX'.
    -- Pair when the bare's normalized_name is a token subset of the
    -- full's normalized_name (via distinctive_tokens() Python
    -- semantics — approximated here by whole-word substring; the
    -- 14 confirmed pairs all satisfy the stronger token-subset test
    -- so the approximation is safe for discovery, and the operator
    -- eyeballs the shortlist).
    SELECT
        bare.id                AS bare_team_id,
        bare.canonical_name    AS bare_canonical,
        bare.normalized_name   AS bare_normalized,
        bare.country_code      AS bare_country_code,
        bare.created_at        AS bare_created_at,
        full_row.id                 AS full_team_id,
        full_row.canonical_name     AS full_canonical,
        full_row.normalized_name    AS full_normalized,
        full_row.country_code       AS full_country_code,
        full_row.created_at         AS full_created_at
    FROM baseball_teams bare
    JOIN baseball_teams full_row
      ON bare.id != full_row.id
     AND full_row.normalized_name LIKE '%' || bare.normalized_name || '%'
     AND full_row.normalized_name != bare.normalized_name
    WHERE bare.created_at::date  = '2026-05-08'
      AND full_row.created_at::date  = '2026-05-28'
      AND bare.country_code IS NULL
      AND full_row.country_code = 'MEX'
)
SELECT
    p.bare_team_id,
    p.bare_canonical,
    p.full_team_id,
    p.full_canonical,
    -- Confirmation column, not a filter:
    (p.bare_country_code IS NULL AND p.full_country_code = 'MEX') AS country_code_discriminator_agrees,
    -- Fixture counts — expect ~21-30 on bare, 0 on full:
    (SELECT count(*) FROM sp.fixtures fx
     WHERE fx.home_team_id = p.bare_team_id OR fx.away_team_id = p.bare_team_id) AS bare_fixture_count,
    (SELECT count(*) FROM sp.fixtures fx
     WHERE fx.home_team_id = p.full_team_id OR fx.away_team_id = p.full_team_id) AS full_fixture_count,
    -- Alias counts — for planning the migration step:
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = p.bare_team_id) AS bare_alias_count,
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = p.full_team_id) AS full_alias_count
FROM substring_pairs p
ORDER BY p.bare_canonical;
```

**Expected**: 14 rows, `country_code_discriminator_agrees = true` on all, `bare_fixture_count` between 21 and 30, `full_fixture_count = 0` on all 14.

**If discovery returns a different row count**: STOP. Investigate whether a new duplicate emerged, or Sultanes/Monterrey pre-reconciliation shifted the count, or the predicate has drifted from the guard.

**If `full_fixture_count > 0` on any row**: STOP. Direction (b) assumes zero fixtures on the full row. Non-zero means fixtures accumulated somewhere unexpected; do NOT delete the full row without a fixture re-point plan (revert to direction (a) for that pair only).

## 7. §3 — Merge sequence per pair (5 steps, ONE transaction per pair)

**Blast-radius discipline**: 14 transactions, NOT one super-transaction. A failure in pair 7 must not block pairs 1-6 from committing or pairs 8-14 from starting cleanly.

Per pair, in this exact order (order matters — the global-alias-conflict check in `bootstrap_lmb.py:232-239` and the FK from `sp.team_aliases.team_id` to `sp.teams.id` both require the sequence below):

```sql
BEGIN;

-- Step 1: RENAME bare row's canonical + normalized to the full club form.
UPDATE sp.teams
SET canonical_name  = '{FULL_CANONICAL}',
    normalized_name = '{FULL_NORMALIZED}',
    country_code    = 'MEX'
WHERE id = '{BARE_TEAM_ID}';
-- Expected: 1 row updated.

-- Step 2: MIGRATE full row's aliases → bare row, but only aliases not
-- already present. Preserves any alias forms attached to the full row
-- (e.g. seed variants) without duplicating what the bare row already has.
INSERT INTO sp.team_aliases
    (id, team_id, alias, alias_normalized, source, confidence, created_at)
SELECT gen_random_uuid(), '{BARE_TEAM_ID}',
       src.alias, src.alias_normalized, src.source, src.confidence, src.created_at
FROM sp.team_aliases src
WHERE src.team_id = '{FULL_TEAM_ID}'
  AND NOT EXISTS (
      SELECT 1 FROM sp.team_aliases dst
      WHERE dst.team_id = '{BARE_TEAM_ID}'
        AND dst.alias_normalized = src.alias_normalized
  );
-- Expected: 0 to N rows inserted; N == full_alias_count minus overlap
-- with bare_alias_count.

-- Step 3: DELETE full row's aliases (all now migrated or already
-- present on the bare row).
DELETE FROM sp.team_aliases WHERE team_id = '{FULL_TEAM_ID}';
-- Expected: rows deleted == full_alias_count (from §1).

-- Step 4: Reference-scan safety check. If the full row is referenced
-- by anything other than sp.team_aliases (already handled), this
-- SELECT surfaces it before step 5 tries to delete. Do NOT proceed
-- to step 5 if this returns non-zero.
SELECT 'fixtures_home' AS ref, count(*) AS n FROM sp.fixtures WHERE home_team_id = '{FULL_TEAM_ID}'
UNION ALL SELECT 'fixtures_away', count(*)     FROM sp.fixtures WHERE away_team_id = '{FULL_TEAM_ID}';
-- Expected: both counts 0 (verified in §1). If non-zero, ROLLBACK
-- and treat this pair as direction (a) territory.

-- Step 5: DELETE full row.
DELETE FROM sp.teams WHERE id = '{FULL_TEAM_ID}';
-- Expected: 1 row deleted.

-- Step 6: Per-pair verification INSIDE the transaction. If any assertion
-- fails, ROLLBACK.
--   6a: Bare row now carries the full canonical.
SELECT canonical_name, normalized_name, country_code
FROM sp.teams WHERE id = '{BARE_TEAM_ID}';
-- Expected: full canonical, full normalized, 'MEX'.
--   6b: Full row is gone.
SELECT count(*) FROM sp.teams WHERE id = '{FULL_TEAM_ID}';
-- Expected: 0.
--   6c: Alias union is intact on the bare row.
SELECT count(*) FROM sp.team_aliases WHERE team_id = '{BARE_TEAM_ID}';
-- Expected: original bare_alias_count + (aliases migrated in step 2).

COMMIT;
```

Repeat for each of the 14 pairs, substituting placeholders per pair. Between pairs, run the §0 pre-flight again (cheap; catches DB-name drift if session moved).

## 9. §5 — Post-merge natural verification (the null-observable guard)

**Two auxiliary premises, both required.** Neither alone makes the observation dispositive; both together do.

### Premise (a) — the FL daily cron ran

```sql
-- Confirm the 02:00 UTC FL daily cron actually fired and processed
-- records. Without this, an empty result in premise (b) is meaningless
-- (the class was never scanned).
SELECT provider, started_at AT TIME ZONE 'UTC' AS started_utc,
       records_scanned, auto_applies, no_match, crashes
FROM sp.resolver_runs
WHERE provider = 'fl'
  AND run_mode = 'cron'
  AND started_at > NOW() - INTERVAL '1 day'
ORDER BY started_at DESC
LIMIT 1;
-- Expected: exactly 1 row, started_utc at ~02:00,
-- records_scanned > 0, crashes = 0.
```

### Premise (b) — at least one of the 14 merged teams was in the scan

```sql
-- Without this, "0 collision failures in the delta window" means
-- either "the class is now healthy" OR "FL sent no LMB fixtures in
-- the delta window for a boring reason (season break, schedule
-- gap)". Requiring at least one of the 14 team_ids to appear in the
-- post-cron resolution_log rules out the boring cause.
WITH merged_team_ids AS (
    SELECT bare_team_id AS team_id FROM sp.lmb_dedup_snapshot_2026_07_19
),
recent_log AS (
    SELECT rl.provider_record_id, rl.reason_code, rl.reason_detail,
           rl.decided_at, rl.resolver_version
    FROM sp.resolution_log rl
    WHERE rl.provider = 'fl'
      AND rl.decided_at > (SELECT MAX(started_at) FROM sp.resolver_runs
                            WHERE provider = 'fl' AND run_mode = 'cron')
),
log_touching_merged AS (
    SELECT rl.*, m.team_id AS matched_merged_team_id
    FROM recent_log rl
    JOIN merged_team_ids m
      ON  (rl.reason_detail->>'home_team_id')::text = m.team_id::text
       OR (rl.reason_detail->>'away_team_id')::text = m.team_id::text
)
SELECT count(*) AS log_rows_touching_merged_teams,
       count(DISTINCT matched_merged_team_id) AS distinct_merged_teams_touched
FROM log_touching_merged;
-- Expected: log_rows_touching_merged_teams > 0
-- AND distinct_merged_teams_touched >= 1.
```

### The verification predicate (dispositive if BOTH premises hold)

```sql
-- Expected post-merge shape, per the Querétaro control:
--   reason_code = 'alias' (NOT 'strict')
--   resolver_version = 'alias@2c.0'
--   reason_detail->>'home_collision' = 'false' (or absent)
--   reason_detail->>'home_ratio' = '1.0' (or very close)
--
-- The bridge is the alias tier's token_set_contribution — the same
-- scorer the Day-48 F8 break watched auto-apply. If a merged record
-- comes back with reason_code='strict', SOMETHING ELSE changed and
-- we should notice — the expected path is alias-tier via canonical
-- token-set, not strict via exact alias hit.

WITH merged_team_ids AS (
    SELECT bare_team_id AS team_id FROM sp.lmb_dedup_snapshot_2026_07_19
),
recent_log AS (
    SELECT rl.provider_record_id, rl.reason_code, rl.reason_detail,
           rl.decided_at, rl.resolver_version
    FROM sp.resolution_log rl
    WHERE rl.provider = 'fl'
      AND rl.decided_at > (SELECT MAX(started_at) FROM sp.resolver_runs
                            WHERE provider = 'fl' AND run_mode = 'cron')
)
SELECT
    rl.provider_record_id,
    rl.reason_code,
    rl.resolver_version,
    rl.reason_detail->>'home_team_id' AS home_team_id,
    rl.reason_detail->>'away_team_id' AS away_team_id,
    rl.reason_detail->>'home_collision' AS home_collision,
    rl.reason_detail->>'away_collision' AS away_collision,
    rl.reason_detail->>'home_ratio' AS home_ratio,
    rl.reason_detail->>'away_ratio' AS away_ratio
FROM recent_log rl
JOIN merged_team_ids m
  ON  (rl.reason_detail->>'home_team_id')::text = m.team_id::text
   OR (rl.reason_detail->>'away_team_id')::text = m.team_id::text
ORDER BY rl.decided_at DESC;
-- Expected on every row touching a merged team_id:
--   reason_code IN ('alias', 'strict')  -- alias expected, strict acceptable if aliases already covered
--   home_collision IN (NULL, 'false')
--   away_collision IN (NULL, 'false')
--   home_ratio / away_ratio == '1.0' where present
--
-- If reason_code = 'review_queue' on any merged-team row: the merge
-- did not close the class. Investigate that record specifically.
-- If reason_code = 'strict' unexpectedly: SOMETHING ELSE changed
-- (alias landed, matcher path shifted, etc.); investigate but do
-- not automatically treat as failure.
```

**Verification passes** iff premises (a) AND (b) hold, and the predicate query returns rows with `reason_code IN ('alias', 'strict')` and no `home_collision:true` / `away_collision:true` on any merged team. If premise (b) fails (no merged team in the delta window), verification is INCOMPLETE, not FAILED — re-run after the next cron until premise (b) is satisfied.

## 11. §7 — Rollback per pair (byte-exact restore from snapshot)

If verification fails on any pair, or a later diagnostic surfaces a bug in the merge sequence, restore is snapshot-driven and byte-exact.

Per pair:

```sql
BEGIN;

-- Step 1: Restore the full row with its original team_id (preserved
-- in the snapshot — the DELETE in §7 removed the row but the id was
-- captured pre-merge).
INSERT INTO sp.teams (id, sport_id, canonical_name, normalized_name, country_code, created_at)
SELECT s.full_team_id,
       (SELECT sport_id FROM sp.teams WHERE id = s.bare_team_id) AS sport_id,
       s.full_canonical_pre,
       s.full_normalized_pre,
       s.full_country_code_pre,
       -- created_at comes from the snapshot payload:
       (s.full_aliases_pre->0->>'created_at')::timestamptz
FROM sp.lmb_dedup_snapshot_2026_07_19 s
WHERE s.pair_id = '{PAIR_ID}';
-- Expected: 1 row inserted.

-- Step 2: Restore full row's aliases from snapshot payload.
INSERT INTO sp.team_aliases (id, team_id, alias, alias_normalized, source, confidence, created_at)
SELECT (a->>'id')::uuid,
       s.full_team_id,
       a->>'alias',
       a->>'alias_normalized',
       a->>'source',
       (a->>'confidence')::numeric,
       (a->>'created_at')::timestamptz
FROM sp.lmb_dedup_snapshot_2026_07_19 s,
     jsonb_array_elements(s.full_aliases_pre) a
WHERE s.pair_id = '{PAIR_ID}';
-- Expected: rows inserted == snapshot's full_aliases_pre count.

-- Step 3: Remove the aliases that were migrated from full → bare in
-- the merge (identified by presence in full_aliases_pre AND absence
-- from bare_aliases_pre; anything present in both was on the bare
-- row already and stays).
DELETE FROM sp.team_aliases
WHERE team_id = (SELECT bare_team_id FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}')
  AND alias_normalized IN (
      SELECT DISTINCT fa->>'alias_normalized'
      FROM sp.lmb_dedup_snapshot_2026_07_19 s,
           jsonb_array_elements(s.full_aliases_pre) fa
      WHERE s.pair_id = '{PAIR_ID}'
        AND NOT EXISTS (
            SELECT 1 FROM jsonb_array_elements(s.bare_aliases_pre) ba
            WHERE ba->>'alias_normalized' = fa->>'alias_normalized'
        )
  );

-- Step 4: Revert bare row's canonical / normalized / country_code.
UPDATE sp.teams
SET canonical_name = (SELECT bare_canonical_pre  FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}'),
    normalized_name = (SELECT bare_normalized_pre FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}'),
    country_code    = (SELECT bare_country_code_pre FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}')
WHERE id = (SELECT bare_team_id FROM sp.lmb_dedup_snapshot_2026_07_19 WHERE pair_id = '{PAIR_ID}');
-- Expected: 1 row updated.

-- Step 5: Verify byte-exact restore (INSIDE the transaction; ROLLBACK
-- if any assertion fails).
SELECT
    (SELECT canonical_name FROM sp.teams WHERE id = s.bare_team_id) = s.bare_canonical_pre AS bare_canonical_restored,
    (SELECT normalized_name FROM sp.teams WHERE id = s.bare_team_id) = s.bare_normalized_pre AS bare_normalized_restored,
    (SELECT count(*) FROM sp.teams WHERE id = s.full_team_id) = 1 AS full_row_restored,
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = s.full_team_id) = jsonb_array_length(s.full_aliases_pre) AS full_aliases_restored,
    (SELECT count(*) FROM sp.team_aliases WHERE team_id = s.bare_team_id) = jsonb_array_length(s.bare_aliases_pre) AS bare_aliases_restored
FROM sp.lmb_dedup_snapshot_2026_07_19 s
WHERE s.pair_id = '{PAIR_ID}';
-- Expected: all five columns TRUE. If any FALSE, ROLLBACK.

COMMIT;
```
