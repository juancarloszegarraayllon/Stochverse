# Re-Resolution Loop — Scope Doc

Phase 2 exit-gate workstream (Day-40, post-BBL strategic fork). The §7.6 / §7.7 re-resolution loop — first Phase 2E workstream, first work on the active critical-path to v4 cutover since the Day-21 measurement-cron debt was opened.

This document is **design only**, no build. Same survey-first → scope-second → build-third discipline that produced Component 4. Mirrors the Phase 2D.5-A scope-doc convention (`docs/dedup/tennis-2026-05-25.md`): F-framing matrix, in/out boundaries, F8 success criterion, rollback plan, risks.

## 1. Context and motivation

### What the architecture asks for

SP Architecture v1.4 §7.7 specifies a **three-loop resolver**:

  - **Hot loop** — sub-second, LISTEN/NOTIFY on new provider rows.
  - **Batch loop** — 30 s safety-net poll.
  - **Re-resolution loop** — 5–10 min sweep of low-confidence + alias-add events.

§7.6 names the re-resolution loop as the back-catalog re-link mechanism: when an alias is added today, the loop retroactively re-resolves every record the resolver previously failed on that the new alias would have helped.

Of the three loops, **none ships today as a loop**. What ships is the daily cron (FL 02:00 UTC + Kalshi 02:15 UTC) — a single pass at the wrong cadence and the wrong trigger semantics for any of the three.

### What's actually in production today (the foundation)

Verified Day-40 survey, not re-derived here:

1. **A crude re-resolution already runs.** Every daily cron pass re-resolves every `fixture_id IS NULL` provider record from scratch (`scripts/run_resolver_pass.py:283` + `:310`). It's `O(unresolved-population)` work twice a day. It catches records that "now resolve" because someone added an alias in the prior 24 hours, but it does so by re-running the matcher against the whole unresolved set rather than the records actually affected.
2. **The matcher works.** `resolver.matcher.TieredMatcher` (strict → alias → fuzzy) is the right thing to call. This workstream does not modify it.
3. **Data model has the bones.** `sp.resolution_log.reason_code` + `sp.resolution_log.decided_at` + `sp.team_aliases.created_at` + the `fixture_id IS NULL` filter all exist and are indexed for the gross filters. The **missing piece** is efficient candidate selection — "which previously-no_match records would now resolve given alias X added at time T?" — because the "almost-hit" signal lives in un-indexed `resolution_log.reason_detail` JSONB.
4. **`'live'` `run_mode` is reserved.** `sp.resolver_runs.run_mode` already accepts `'live'` (`sp_models.py:501`); `scripts/run_resolver_pass.py:149` rejects it pending this workstream.

### Why this is the right next workstream

Recorded in PROJECT_STATE Day-40 (this doc supports the same case; not re-litigating). One-line restatement: the loop **compounds across all 39 days of prior coverage** AND unlocks the 18,303-row review queue as a re-sweep input. One workstream, two Phase 2 exit gates progressed. The next league bootstrap moves neither.

### Operator decisions already made (Day-40)

Two decisions go in here as inputs, not framing questions:

- **Cadence: POLL-FIRST.** 5–10 min cron, same pattern as the FL/Kalshi crons. Not LISTEN/NOTIFY. Escalation trigger: `DEPLOYMENT.md:1152` — if p95 latency > 5 min sustained, escalate to LISTEN/NOTIFY at Phase 2E.fix. Reason: the loop's value is the accuracy multiplier (back-catalog re-resolution), which a poll collects fully; LISTEN/NOTIFY buys latency that only matters for Phase 3+ serving. Don't build real-time before there's a real-time product.
- **Daily-diff cron wiring folded in.** Same `railway.toml` surface as the loop. The loop is unmeasurable without the diff; closes the Day-21 measurement-gap debt. Two cron entries, one workstream.

## 2. Scope boundaries

### In scope

- `scripts/run_reresolution_pass.py` — new script, mirrors `scripts/run_resolver_pass.py` shape (bulk-fetch candidate set, run `TieredMatcher`, write `sp.resolution_log`, write `sp.resolver_runs` row with `run_mode='live'`, structlog events, halt-criteria warnings, Pattern D pre-flight on the write path).
- **Candidate-selection layer** — the targeted query that makes this loop fundamentally different from the daily cron's brute-force re-resolve. Designed in §3 F1.
- One alembic migration — index strategy for the candidate-selection query's hot filter. Designed in §3 F2.
- `railway.toml` — two new cron service entries: re-resolution loop + daily-diff. Cadence + stagger designed in §3 F3.
- Idempotency confirmation — the existing `ON CONFLICT (provider, provider_record_id) DO UPDATE WHERE status='pending'` pattern at `run_resolver_pass.py:640` must cover the re-resolution write path with no changes. Verified in §3 F4.
- Observability + halt criteria — mirror what already works in `run_resolver_pass.py` (per-pass run_id, sp.resolver_runs row, structlog events, halt-criteria stdout for cron-log scrapers). Designed in §3 F6.
- F8 success criterion — designed in §3 F7; the loop must be measurably effective on the back-catalog before it's left running.

### Out of scope (named so they're not pulled in)

- **Hot LISTEN/NOTIFY loop.** Phase 2E.fix per `DEPLOYMENT.md:1152`. Only if p95 > 5 min sustained.
- **§6.5 archival job.** Separate Phase 2 exit gate. Orthogonal; deserves its own scope doc.
- **Review-queue harvest tooling.** Downstream of the loop. The loop drains *machine-resolvable* queue rows whose backing aliases are already in place; harvesting operator decisions back into `sp.team_aliases` is a separate workstream.
- **Matcher tuning.** `TieredMatcher` ships as-is. If the loop surfaces a new failure mode, that's a Phase 2D follow-on, not part of this scope.
- **`/api/v4` cutover scaffolding.** Phase 3. Still gated behind queue health + measurement.

### What the loop is, in one line

A scheduled poll that, every 5–10 minutes, identifies the small subset of previously-no_match provider records whose underlying signals or aliases have changed since their last decision, re-runs the matcher against just that subset, and emits a fresh `sp.resolution_log` row per record.

### F1 — Candidate-selection query (THE core design question)

**Question:** Which previously-no_match records get re-processed each pass?

This is the single design choice that distinguishes the loop from the daily cron's brute force. Get this right, the loop is cheap and effective. Get it wrong, the loop is either expensive (selects too many records) or ineffective (misses the ones that would now resolve).

**Default: target-set = records whose latest no_match decision predates a relevant alias-add OR a relevant fixture-state change.**

Formally, candidate `(provider, provider_record_id)` is included iff all three conditions hold:

1. **Still unresolved.** `provider_table.fixture_id IS NULL`. Same filter the daily cron uses.
2. **Last decision was no_match.** The most recent `sp.resolution_log` row for `(provider, provider_record_id)` has `reason_code = 'no_match'`. (`reason_code` is one of `strict`, `alias`, `fuzzy`, `review_queue`, `no_match`, `crash`, `signal_extraction_skipped` — only `no_match` is loop-eligible; `review_queue` is operator-blocked and `crash` needs investigation.)
3. **Something relevant has changed since that decision.** At least one of:
   - **Alias-add signal**: a new `sp.team_aliases` row exists with `created_at > last_decision.decided_at`, AND the alias's `team_id` or `alias_normalized` appears in the prior decision's `reason_detail` (the "almost-hit" signal — strict-tier emits the colliding-team-ids and the failed-side trigram candidates per `resolver/fuzzy_tier/matcher.py:521-538`; alias-tier emits the canonical name the matcher tried; the previously-considered teams that have now received a new alias are the population worth re-running).
   - **Fixture-state signal**: a new sp.fixtures row exists with `created_at > last_decision.decided_at` whose home/away team pair overlaps the prior decision's candidate set. Catches Day-N+1 corroboration cases — Kalshi record couldn't strict-match because FL hadn't ingested the fixture yet; today FL did; re-resolve now picks it up. Per `PHASE_2D_DESIGN.md:262` this is the primary mechanism by which the existing daily cron already lifts records on retry; the loop just does it within minutes instead of a day.

**Why this matters:** the daily cron walks ~thousands of records per pass; targeted selection walks ~tens-to-hundreds. With a 5-min cadence the loop runs 288×/day vs the cron's 2×/day; without targeting, that's 144× the cost for the same work. Targeted, the work is proportional to actual change.

**Open sub-question for operator** (F1a): is the alias-add signal strict (the new alias's `(team_id, alias_normalized)` must literally appear in the prior `reason_detail`'s colliding_team_ids / candidate_team_ids list) or loose (any new alias on any of the prior decision's candidate teams)?

- **Strict** is precise but misses cases where the matcher's candidate set was narrower than reality (e.g. strict tier surfaced one collision pair but a new alias on a third team would have made it pick that team). Cheaper to compute.
- **Loose** is over-broad — it re-runs records whose alias addition can't help — but the matcher itself is cheap once the candidate set is small, and the false-positive re-runs just write another `no_match` row.

Default proposed: **loose**. Cheaper to design, and the matcher's existing decision logic is the right re-classifier. Operator override if measured cost surprises us.

### F2 — Index strategy for the candidate-selection query

**Question:** F1's condition (3) is the only non-trivial join — `reason_detail` JSONB is currently un-indexed. GIN index on the whole `reason_detail` column, or extract specific keys into typed columns?

**Default: GIN index on `sp.resolution_log.reason_detail` with `jsonb_path_ops` operator class.** Single alembic migration. Targets the path-existence queries the loop's condition (3) wants (`reason_detail @> '{"colliding_home_team_ids": ["…"]}'` shape).

Rationale:

- **GIN with `jsonb_path_ops`** is the right index class for containment queries on JSONB. Smaller than the default `jsonb_ops`, faster lookup, only supports `@>` (which is exactly what condition 3 needs — "does this reason_detail mention this team_id?").
- **Extracting columns** (e.g., a separate `sp.resolution_log_team_refs` join table populated by trigger) would be faster per-lookup but adds write-path mechanism on every resolution_log INSERT — bad trade given the loop runs every 5 min and the cron runs every 12 hours. The JSONB INSERT path stays unchanged; only the read pattern changes.
- **Hybrid (extract only the team_id arrays)** is plausible if measurement shows GIN is too slow. Defer until measured. Same shape Tennis dedup deferred its data-model questions until production runs surfaced them (`docs/dedup/tennis-2026-05-25.md` §3 F4).

The migration shape:

```python
op.execute(
    "CREATE INDEX CONCURRENTLY ix_resolution_log_reason_detail_gin "
    "ON sp.resolution_log USING gin (reason_detail jsonb_path_ops)"
)
```

`CREATE INDEX CONCURRENTLY` is required because `sp.resolution_log` is the production hot-write table (130k+ rows per Day-37 measurement; the daily cron writes ~5-15k/day). Non-concurrent CREATE INDEX would lock the table for the duration. CONCURRENTLY needs care in alembic (no transaction wrap — `op.execute` with autocommit, or split out of any wrapped revision). Operator-deliberate run, like the manual `alembic upgrade head` discipline in `DEPLOYMENT.md`.

**Sub-question for operator** (F2a): does the operator want to pre-measure the GIN selectivity on production before the migration ships, or accept the design risk and measure post-migration via daily-diff? Default proposed: **accept post-migration measurement**; if GIN underperforms, the alembic migration's downgrade is `DROP INDEX CONCURRENTLY` (also non-blocking).

### F3 — Cadence + stagger

**Question:** 5 min or 10 min? Stagger when?

**Default: 5 min, staggered :02 / :07 / :12 / … on the FL service.** Per-provider crons mirror the existing FL/Kalshi separation (`railway.toml` runs them at 02:00 / 02:15 UTC daily). The Day-21 swap that made FL go first (so Kalshi sees fresh fixtures for corroboration) applies here too: FL loop ahead of Kalshi loop by ~2-3 minutes minimum.

Proposed `railway.toml` additions:

```toml
[[services]]
name = "resolver-reresolution-fl"

[services.deploy]
cronSchedule = "*/5 * * * *"
startCommand = "python scripts/run_reresolution_pass.py --provider fl --run-mode live"
restartPolicyType = "NEVER"

[[services]]
name = "resolver-reresolution-kalshi"

[services.deploy]
# Offset by 2 minutes so Kalshi's pass sees FL's just-resolved fixtures.
cronSchedule = "2-59/5 * * * *"
startCommand = "python scripts/run_reresolution_pass.py --provider kalshi --run-mode live"
restartPolicyType = "NEVER"

[[services]]
name = "daily-diff"

[services.deploy]
# Daily at 03:00 UTC, 45 min after the daily cron pair. Folds in the
# Day-21 measurement-debt — the diff is the loop's measurement.
cronSchedule = "0 3 * * *"
startCommand = "python scripts/daily_diff.py"
restartPolicyType = "NEVER"
```

**Why 5 min, not 10**: at 5 min the per-pass cost is fully bounded by the candidate-selection query's selectivity (which targeting makes small), and the latency on a newly-added alias hitting the back-catalog is ≤ 5 min. At 10 min the cost is the same per-pass; the only thing that changes is alias-add latency goes up. No reason not to pick 5 if the cost holds.

**Risk**: cron drift on Railway. Railway crons aren't precisely scheduled (typical ±30s). The 2-minute FL/Kalshi offset is comfortable margin; the 45-min daily-diff offset is generous.

**Sub-question for operator** (F3a): does the operator want the loop quiet during the daily cron windows (02:00 / 02:15 UTC) to avoid overlapping work? The daily cron also writes to `sp.resolver_runs` with `run_mode='cron'`; the loop writes `run_mode='live'`. Both run independently per `sp.resolver_runs.run_mode` semantics. Default proposed: **no quiet window** — the loop's candidate-selection naturally excludes records the daily cron just resolved (their latest decision will not be no_match for long; targeting filters them).

### F4 — Idempotency / write-path safety

**Question:** Does the existing `ON CONFLICT (provider, provider_record_id) DO UPDATE WHERE status='pending'` pattern at `run_resolver_pass.py:640` cover the re-resolution write path with no changes?

**Default: yes, no changes needed.** Verified:

- `sp.review_queue` writes happen on the REVIEW_QUEUE-shaped matcher decisions. The existing ON CONFLICT clause was added in Phase 2D.3.1 specifically because *"the same unresolved record (fixture_id IS NULL) comes back to the resolver on every cron pass until an operator approves the review_queue row, so the second pass would otherwise IntegrityError"* (line 583-588). That precondition is exactly the re-resolution loop's design — records that didn't strict-resolve come back. The clause handles it.
- `WHERE status='pending'` (line 589-595) — operator decisions (approve / reject) are preserved across re-runs. Loop sees an operator-rejected row and writes an updated reason_detail but leaves `status` alone.
- `sp.resolution_log` is append-only. Each pass writes a fresh row. Indexed on `(provider, provider_record_id)` + `decided_at`. The loop's writes look identical to the cron's writes.
- `sp.resolver_runs` carries `run_id` + `run_mode` distinguishing each pass. Loop emits `run_mode='live'`.

**Sub-question for operator** (F4a): the daily cron's `run_mode='cron'` rows are how the operator measures pass-level throughput today. Should `run_mode='live'` rows show up in the same `sp.resolver_runs` view, or be filtered separately? Default proposed: **same view, filter by `run_mode` when desired**. The Phase 2B `ix_resolver_runs_run_mode_started` index supports both reading patterns.

### F6 — Halt criteria + observability

**Question:** When should the loop log a halt-criteria warning vs. exit non-zero (deploy alert)?

**Default: mirror `run_resolver_pass.py` exactly.**

- Halt warnings to stdout AND structured log on:
  - Candidate-selection query latency > 5 s (we're at design budget; investigate before tuning).
  - Per-pass record count > 5× the trailing 7-day mean (something upstream changed; investigate).
  - GIN index scan ratio drops below some threshold (operator-chosen, e.g., 80%) — index isn't being used.
- Exit code stays 0 on warnings (don't deploy-fail on transient signal). The operator reviews warnings via a periodic query, same as the daily cron's halt-criteria pattern (`run_resolver_pass.py:127` comment).
- Exit non-zero only on hard failures (DB unreachable, Pattern D mismatch, signature error in `TieredMatcher`).

Structured log fields per pass: `run_id`, `provider`, `run_mode='live'`, `candidate_set_size`, `auto_applies`, `no_match`, `review_queue_writes`, `crashes`, `latency_total_ms`, `latency_candidate_select_ms`.

**Sub-question for operator** (F6a): should the loop's `sp.resolver_runs` rows be retained forever (like the cron's), or pruned after some window? Defaults aligned with `sp.resolution_log` (retained forever per §6.5). Default proposed: **retained forever**.

### F7 — Success criterion + F8 validation (the headline)

**Question:** How do we F8-validate the loop is working before leaving it running?

**Default: two-part validation, mirrors the Component 4 "verify every seam against the new domain" discipline.**

**Part A — Targeted before/after (the dispositive check):**

Pick a known set of records that a recent alias-add SHOULD now resolve. Operator stages this from production:

1. Identify an alias added in the last 24h (any provenance).
2. Run a one-off query for `(provider, provider_record_id)` rows where the prior decision was no_match AND `reason_detail` mentions the team that just received the alias.
3. Take an empirical snapshot of those records pre-loop.
4. Trigger one loop pass; let it run.
5. Snapshot the same records post-loop.

**Pass criterion**: every record in the snapshot that the loop's candidate-selection caught now has a fresh `sp.resolution_log` row, and the new row's `reason_code` is either `strict`/`alias`/`fuzzy` (resolved) or `review_queue` (operator review needed) or still `no_match` (genuinely unmatched). The dispositive negative check: zero records *should have been* caught by the candidate selection but weren't. Test the candidate-selection query directly.

**Part B — Aggregate capability trajectory (the leverage check):**

Daily-diff `aggregate matcher_capability_rate` is the existing measurement (Day-15 baseline 35.3% on 2026-06-15). The loop's value is the rate climbing as the back-catalog re-sweeps. Loop wired Day-N → measure Day-N+1 through Day-N+7. Per-sport drift via the existing `daily_diff` per-sport breakdown (`PROJECT_STATE.md:53`).

**Pass criterion**: by Day-N+7, aggregate matcher capability is up by at least the back-catalog lift the operator expects from the recent alias-add population (operator estimates the expected lift before the loop ships, then measures). No specific numeric threshold imposed; the criterion is "did the loop deliver the lift we expected from the recent additions, or did it underperform / overperform unexpectedly."

**Part C — Negative check (no regression on the daily cron's work):**

Confirm the daily cron continues to write `run_mode='cron'` rows at the existing cadence and that those rows show the same `auto_applies` / `no_match` distribution as pre-loop. The loop and the daily cron are complementary, not competing. If the loop is starving the daily cron of work (because everything got pre-resolved by the loop), that's a feature; if the loop is *crashing* the daily cron (e.g. lock contention), that's a regression.

### F8 — First-pass scope + future seams

**Question:** What does the operator NOT have to commit to today?

**Default: keep the seam, defer the scaling pieces.**

- **LISTEN/NOTIFY escalation**: the candidate-selection query is the same shape whether triggered by poll or by NOTIFY. The `scripts/run_reresolution_pass.py` entry point will accept a `--candidate-set <provider,record_id,…>` mode (one record at a time, no candidate-selection query) so a future LISTEN handler can shovel events in without rewriting the pipeline. Designed-in seam, not built.
- **Per-sport tuning**: the loop runs sport-agnostic. If one sport's candidate-selection produces an outsized population (e.g., Tennis surname workstream interactions), the operator can fork sport-specific cron entries by adding `--sport <id>` to the script. The script accepts the arg from day one; the loop ships running all sports.
- **Re-resolution of `review_queue` rows on operator approval**: when an operator approves a review_queue row, the matcher's decision is preserved (per F4) but the alias backing it may now help other records. That's the harvest workstream — explicit out of scope per §2.
- **Re-running on Phase 2D fuzzy-tier changes**: today the fuzzy tier is the third tier. If Phase 2D tuning lands a new fuzzy strategy, the loop will pick it up automatically (it calls TieredMatcher which is the version-stamped pipeline). No additional design.

- Single SQL query in `scripts/run_reresolution_pass.py`.

## 5. Success criteria + measurement plan

Per F7:

- **F7 Part A**: targeted before/after on a known alias-add population. Pass = candidate selection catches the right records.
- **F7 Part B**: aggregate matcher capability rises in line with operator's pre-loop estimate over Day-N+1 through Day-N+7.
- **F7 Part C**: daily cron unchanged in distribution (no regression).

Measurement vehicle: the daily-diff cron just wired. Aggregate matcher capability + per-sport breakdown + review_queue depth all flow through `scripts/daily_diff.py` and land in `sp.daily_diff_reports`. The Day-21 debt closes the same day the loop ships.

## 6. Rollback procedure

Two layers:

**Loop misbehaving** (excessive candidate set, wrong matches, lock contention):
1. Disable both `resolver-reresolution-*` services in the Railway dashboard. Cron stops; no code revert needed.
2. Daily cron continues to handle the unresolved population (the pre-loop steady state).
3. Investigate. If the candidate-selection query is wrong, fix forward (the loop's outputs are append-only `sp.resolution_log` rows; nothing to undo).
4. Re-enable when fixed.

**GIN index misbehaving** (excessive write amplification on resolution_log INSERTs):
1. `DROP INDEX CONCURRENTLY ix_resolution_log_reason_detail_gin`. Online, non-blocking.
2. Loop continues but with sequential scans; performance degrades, candidate-selection latency rises. Halt-criteria fires.
3. Disable the loop per the above. Daily cron resumes the load.

**Daily-diff cron misbehaving** (write failures into `sp.daily_diff_reports`):
1. Disable the `daily-diff` service. Reverts to Day-21 state.
2. No data loss — `sp.daily_diff_reports` is the measurement substrate, not load-bearing on the resolver pipeline.

No data-corruption risk on any of the three failure modes. The loop's writes are append-only to `sp.resolution_log`; the migration is online; the cron service entries are independently toggleable.

## 7. Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Candidate-selection query is too broad → loop walks the world | medium | LIMIT on the candidate query; halt-criteria on candidate_set_size > 5× trailing mean; operator-tunable threshold |
| GIN index too slow on production reason_detail volume (130k+ rows) | medium | `jsonb_path_ops` operator class is the smaller / faster choice; if still slow, hybrid extract-team_ids column as a follow-on (designed-in deferral per F2) |
| Loop and daily cron lock-contend on `sp.resolution_log` writes | low | Append-only writes; FK on `fixture_id` is the only cross-table FK; both processes write small chunks |
| Cron cadence drift on Railway (≥ 5 min gaps) | low | 5-min cadence has 2 passes of buffer at 10 min absolute worst; halt-criteria catches sustained drift via latency warning |
| Daily-diff cron write to `sp.daily_diff_reports` collides with a manual run | low | Idempotency by `event_date` UNIQUE constraint (verify in the cron-wiring step; if absent, ADD it) |
| LISTEN/NOTIFY pressure builds before we escalate | low | p95 latency monitor per `DEPLOYMENT.md:1152`; escalate to Phase 2E.fix when triggered |
| F1's "loose" alias-add signal over-runs records that can't help | medium | Matcher re-classification is cheap; over-runs write a fresh no_match row, no harm; F1a sub-question lets operator switch to strict if measured cost surprises |
