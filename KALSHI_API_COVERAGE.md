## Strategic summary

Kalshi exposes a much wider surface than we touch today —
most of it is the trading / portfolio side, which we don't
light up because we don't place orders on behalf of users.
The market-data bucket is what powers /sports, event pages,
and price charts; that's where coverage actually matters.

| Bucket | Coverage | Status |
|---|---|---|
| **Market data (events, markets, history, live)** | 7/33 = 21% | 🚧 The surface our product sits on. Gaps here directly limit what /sports + event pages can show. |
| **Portfolio (positions, orders, fills, account)** | 0/43 = 0% | 🚧 Trading flow not in scope until users place orders through Stochverse. Reserved for post-launch. |
| **Communications (announcements, FCM)** | 0/13 = 0% | 🚧 Separate notification product. Deferred. |
| **Meta (exchange status, search)** | 0/7 = 0% | 🚧 Small utility surface; pick endpoints up as needed (status banner, etc.). |

## WebSocket channels reached today

Subscribed via `_subscribe_batch()` in `kalshi_ws.py`.
Endpoint: `wss://api.elections.kalshi.com/trade-api/ws/v2`.

| Channel | Use |
|---|---|
| `ticker` | Per-market last price + volume — drives every live YES/NO chip on /sports and event pages. |
| `orderbook_delta` | On-demand when an event modal opens the order-book panel; ref-counted unsubscribe when the last browser client disconnects. |
| `trade` | On-demand same as orderbook_delta — feeds the live trades tape on the event page. |

## Gap analysis

### Market-data gaps (product-relevant)

These are the endpoints worth evaluating for the next round
of /sports + event-page features:

| Endpoint | Why it matters | Status |
|---|---|---|
| `GET /events/{event_ticker}` | Single-event detail — richer than what's in our cache record. Useful for the modal's metadata block. | Cache covers most fields today; evaluate when modal needs more. |
| `GET /events/{event_ticker}/metadata` | Settlement source + rules text. Currently surfaced from cache where present, but spec exposes a dedicated endpoint. | Pick up if cache lacks fields we want. |
| `GET /series/{series_ticker}` | Series-level metadata (category, frequency). Already covered via SDK `client.get_series()` for warm-loop sport classification. | ✅ Covered. |
| `GET /markets/candlesticks` historical bulk | Already covered for the per-event chart via `/series/{}/markets/{}/candlesticks`. | ✅ Covered. |

### Portfolio gaps (trading flow)

Light up when implementing the user-trading path. Until
then these are intentionally uncovered — they require
user-scoped API credentials and order-management UI.

### Communications + meta

- `GET /exchange/status` is a one-line health check we
  could surface in a status banner — ~5 min to wire up if
  we want a 'Kalshi is degraded' UX.
- `GET /search/series` could replace some of our manual
  series-prefix mapping in `_SPORT_SERIES`, but the manual
  mapping is required anyway for the sport bucketing logic.
