## 1. Cache record schema

Source: `python3 kalshi_probe/probe_cache_schema.py` →
`/api/_debug/cache_schema`. See full field list there.

**Stable, universally-populated fields** (100% across all sports):

| Field | Type | Meaning |
|---|---|---|
| `event_ticker` | str | Kalshi event ticker (the join key) |
| `series_ticker` | str | Kalshi series — e.g. `KXEPLGAME` |
| `title` | str | Human-readable, e.g. `"Bayern Munich vs PSG"` |
| `category` | str | Kalshi top-level category (`Sports`, `World`, etc.) |
| `outcomes` | list | Per-outcome `{label, prob, yes, no, ticker}` |
| `_sport` | str | Set by our cache builder, e.g. `Soccer` |
| `_close_dt` / `_exp_dt` | str (ISO) | Market close / expiration |

**Key fact**: `_live_state` is **not** stored on cache records.
`/api/events` computes it per request via `match_game` +
`_enrich_soccer_aggregate`. Anything that needs aggregate / series /
display_clock fields must run the same enrichment chain.

## 2. Series tickers per sport

Source: `probe_series.py` → `/api/_debug/series_tickers`.

Series tickers split by suffix into per-fixture markets and
season-long outrights. Suffixes observed in the wild:

```
GAME, MATCH       — per-fixture h2h headline
1H                — first-half winner (soccer only)
F5                — first 5 innings (baseball only)
TOTAL             — totals / over-under
SPREAD            — spread / handicap
BTTS              — both teams to score (soccer only)
1Q, 2Q, 3Q, 4Q    — quarter markets (basketball)
2H                — second-half markets (basketball)
OVERTIME          — binary OT market
ADVANCE           — two-leg tie advancement (soccer cups)
CORNERS           — total corners (soccer)
TCORNERS          — team corners (soccer)
HALFTIME          — halftime score
""  (no suffix)   — outright / future / season-long / props
```

**Suffix is a deterministic predictor of outcome shape** (see §4).

## 3. Title shapes per sport

Source: `probe_titles.py` → `/api/_debug/title_shapes`.

**Universal title shapes**:

| Shape | Meaning | Used by |
|---|---|---|
| `"X vs Y"` | h2h matchup, left=home, right=away | GAME headline (Soccer / NFL / UFL / MLB) |
| `"X at Y"` | h2h matchup, left=away, right=home | TOTAL/SPREAD/BTTS/RFI/F5/etc. (NBA, MLB, soccer sub-markets) |
| `"X vs Y: <market>"` | sub-market on h2h fixture | 1H, ADVANCE, CORNERS, TCORNERS, OVERTIME |
| `"Game N: X at Y"` | playoff series leg | NBA / NHL playoff games |
| `"Series <Type>: A (s) vs B (s)"` | NBA / NHL series-level market | KXNBASERIES, KXNHLSERIES |
| `"<Team> Total Wins"` | per-team season win futures | KXMLBWINS-{TEAM}, KXNFLWINS via market_type |
| free-form question | outrights / futures / yes-no | everything else (no suffix) |

**Title preposition encodes orientation** (universal):

| Preposition | Meaning |
|---|---|
| `vs` | left = home, right = away |
| `at` | left = **away** (visiting), right = home |

US-sports convention: `at` always means visiting team plays at home team's venue.

## 4. Outcome shapes (per sport × suffix)

The renderer's per-shape rules. **Locked from observed data** — no guessing.

### Soccer

| Suffix | Outcome count | Labels | Ties? |
|---|---|---|---|
| `GAME` / `MATCH` | 3 | `home_name`, `away_name`, `Tie` | ✅ |
| `1H` | 3 | same as GAME | ✅ |
| `TOTAL` | 4 | `Over N goals scored` (N=1.5..4.5) | n/a |
| `SPREAD` | 4 | `<team> wins by over N goals` (per team × thresholds) | n/a |
| `BTTS` | 1 | `Both Teams To Score` (binary YES/NO) | n/a |
| `CORNERS` | 5 | `N+ corners` (N=8..12) | n/a |
| `TCORNERS` | 2 | `<team>: N+` per team | n/a |
| `ADVANCE` | 2 | `<team>` (two-leg cup tie qualifier) | n/a |

### Basketball

| Suffix | Outcome count | Labels | Notes |
|---|---|---|---|
| `GAME` | 2 | `home_name`, `away_name` | **No tie** |
| `1H` `2H` (winner) | 3 | `<team> wins 1st half`, `<team> wins 2nd half`, `Tie` | Tie exists for halves |
| `1H TOTAL` | 9 | `Over N 1H points scored` | |
| `1H SPREAD` | 11 | `<team> wins the 1H by over N` | |
| `2H TOTAL` | 9 | analogous | |
| `2H SPREAD` | 11 | analogous | |
| `TOTAL` | 11 | `Over N points scored` | |
| `SPREAD` | 11 | `<team> wins by over N points` | |
| `OVERTIME` | 1 | `Game goes to overtime` (binary) | |
| `MENTION` | 25 | announcer-mention props (NBAMENTION) | |
| Player props (`PTS`, `REB`, `AST`, `BLK`, `STL`, `3PT`, `2D`, `3D`) | 12–50 | `<player>: N+` thresholds | |
| Series (`KXNBASERIES{,SCORE,GAMES,SPREAD,ROADWIN}`) | 2–8 | varies — see §5 | |

### Hockey (NHL)

| Suffix | Outcome count | Labels |
|---|---|---|
| `GAME` | 2 | team labels (NHL: `ANA Ducks`, `VGK Golden Knights` — abbr+name fused) |
| `TOTAL` | 8 | `Over N goals scored` (N=2.5..9.5) |
| `SPREAD` | 4 | `<team> wins by over N goals` |
| `OVERTIME` | 1 | binary `Game goes to overtime` |
| Player props (`PTS`, `GOAL`, `AST`, `FIRSTGOAL`) | 28–67 | `<player>: N+` thresholds |
| Series (`KXNHLSERIES{,SCORE,GAMES,SPREAD}`) | 2–8 | parallel to NBA series structure |

### Baseball (MLB)

| Suffix | Outcome count | Labels |
|---|---|---|
| `GAME` | 2 | team labels (e.g. `Pittsburgh`, `Arizona`) |
| `TOTAL` | 11 | `Over N runs scored` (N=3.5..13.5) |
| `SPREAD` | 6 | `<team> wins by over N runs` |
| `RFI` (KXMLBRFI) | 1 | binary YES — first inning run |
| `F5` (winner) | 3 | `<team> wins first 5 innings`, `Tie` (5-inning split) |
| `F5 TOTAL` | 7 | `Over N runs in the first 5 innings` |
| `F5 SPREAD` | 4 | `<team> -N first 5 innings` |
| `TEAM TOTAL` (KXMLBTEAM) | 14 | `<team> over N runs scored` (per-team × thresholds) |
| Player props (`HIT`, `HR`, `KS`, `TB`, `HRR`) | 9–94 | `<player>: N+` thresholds |
| Per-team season wins (KXMLBWINS-{TEAM}) | 7 | `N+ wins` |

### Football (NFL/UFL)

| Suffix | Outcome count | Labels |
|---|---|---|
| `GAME` | 2 | team labels |
| `TOTAL` (UFL) | 5 | `Over N points scored` |
| `SPREAD` (UFL) | 11 | `<team> wins by over N points` |
| Per-team season wins (KXNFLWINS, market_type=`<team> Total Wins`) | 17 | `N+ wins` |

### Tennis

| Suffix | market_type | Outcomes | Labels |
|---|---|---|---|
| `MATCH` | — | 2 | player full names (NO TIE — tennis can't tie) |
| `SETWINNER` (KXATPSETWINNER, KXWTASETWINNER) | `Set 1 Winner` / `Set 2 Winner` | 2 | player full names |
| `GRANDSLAM` (per-player, KXGRANDSLAM) | `Grand Slam wins in 2026` | 3 | `1+`, `2+`, `3+`, `All 4 Grand Slam wins` |
| Outright tour winners (KXFOMEN, KXFOWOMEN, KXATPGRANDSLAM, KXWTAGRANDSLAM) | — | 14–25 | player full names |

**Tennis sub-tiers** (all use the same `MATCH` shape, different bases for tour level):
- `KXATP` — men's tour (12 records)
- `KXATPCHALLENGER` — men's challenger tier (19 records)
- `KXWTA` — women's tour (34 records)
- `KXWTACHALLENGER` — women's challenger (18 records)
- `KXITF` — men's ITF lower tier (60 records)
- `KXITFW` — women's ITF lower tier (133 records — biggest tennis bucket)

**Title-shape quirk**: tennis uses **surname-only** in `MATCH` titles (`"Hooper vs Ponchet"`) but **full names** in `SETWINNER` titles (`"Taylor Townsend vs Victoria Jimenez Kasintseva: Set 1 Winner"`). Same player pair, two different title shapes depending on suffix. **The renderer should always read player labels from `outcomes[].label`** (which always has the full name), not from titles.

### Esports

Esports uses **multi-map series structure** — a "match" is best-of-N maps, each map is its own betable market. Six games covered: League of Legends, CS2, Valorant, Dota 2, Rocket League, Overwatch, Rainbow Six.

| Suffix / market_type | Outcome count | Labels | Title shape |
|---|---|---|---|
| `GAME` (KXLOL, KXCS2, KXVALORANT, KXDOTA2) | 2 | team names, **no tie** | `"X vs. Y"` (note period after `vs`) |
| `MAP` w/ market_type `Map 1` / `Map 2` / `Map 3` (KXLOLMAP, KXCS2MAP, KXVALORANTMAP, KXDOTA2MAP) | 2 | team names | `"X vs. Y: Map N"` |
| `TOTALMAPS` w/ market_type `Total Maps` (KXLOLTOTALMAPS, KXCS2TOTALMAPS) | 1–2 | `Over N.5 maps` | `"X vs. Y: Total Maps"` |
| Outright (event champion: KXCS2, KXOVERWATCH, KXR6, KXROCKETLEAGUE) | 6–20 | team names | event-name title (e.g. `"IEM Atlanta Champion"`) |

**Note on title format**: Esports uses `"X vs. Y"` (period after `vs`) where other sports use `"X vs Y"` (no period). The title-prep parser must accept both forms.

**Cache-builder bug to flag**: `KXOWGRRANK` (Overwatch GR Rank) is bucketed into Esports but its outcomes are golf-player names (`Scottie Scheffler`, `Rory McIlroy`, etc.). The classifier confused `OWGR` with `OW` (Overwatch). Non-blocking; clean up in a future pass.

### MMA (UFC)

MMA has the **most granular per-fight sub-market structure** of any sport — five distinct sub-markets attached to each headline fight, all sharing the same fixture identity tuple `(date, fighter_abbrs)`.

| Series base | market_type | Outcome count | Label shape |
|---|---|---|---|
| `KXUFCFIGHT` | — | 2 | fighter names, **no draw** in headline. Title: `"<Event>: X vs Y"` |
| `KXUFCDISTANCE` | `Go the Distance` | 1 | binary YES/NO (`Fight goes the distance`) |
| `KXUFCROUNDS` | `Round of Finish` | 2–4 | `Fight ends before round N` |
| `KXUFCVICROUND` | `Round of Victory` | 7–11 | `<Fighter> to win in Round N` × 2 fighters × N rounds + `Decision / Draw / No Contest` |
| `KXUFCMOV` | `Method of Victory` | 7 | `<Fighter> by KO/TKO/DQ` / `by Submission` / `by Decision` × 2 fighters + `Draw` |
| `KXUFCMOF` | `Method of Finish` | 4 | `KO/TKO/DQ`, `Submission`, `Decision`, `Draw` (fighter-agnostic) |

**Critical insight**: `KXUFCROUNDS` ("Round of Finish") and `KXUFCVICROUND` ("Round of Victory") look similar but answer different questions. ROUNDS = "when does the fight end?" (round-agnostic); VICROUND = "who wins in which round?" (per-fighter). Renderer must distinguish them by `market_type`, not by series_base alone.

**Outright title-belt holders** (one per UFC weight class):
`KXUFCBANTAMWEIGHTTITLE`, `KXUFCFEATHERWEIGHTTITLE`, `KXUFCFLYWEIGHTTITLE`, `KXUFCLIGHTWEIGHTTITLE`, `KXUFCLHEAVYWEIGHTTITLE`, `KXUFCHEAVYWEIGHTTITLE`, `KXUFCMIDDLEWEIGHTTITLE`, `KXUFCWELTERWEIGHTTITLE` — 8–11 fighters each. Title: `"UFC <Class> Title Holder on Dec 31, 2026?"`

**Other futures**: `KXMCGREGORFIGHTNEXT` (14 fighters), `KXCARDPRESENCEUFCWH` (5 fighters), `KXUFCWHITEHOUSE` (binary), `KXPERSONUNRETIRE` (binary).

### Golf

Golf is **tournament-based** — fundamentally different from team sports. **No per-fixture h2h ticker exists**; each tournament is its own scoping unit, with up to 12 sub-markets attaching to the same `(tournament_handle, year)` identity.

**Per-tournament markets** (each present for every active tournament — PGA Championship, Truist Championship, ONEflight Myrtle Beach Classic in our sample):

| Series base | market_type | Outcomes | Notes |
|---|---|---|---|
| `KXPGATOUR` | — | 74–131 | Tournament Winner (full field) |
| `KXPGAR1LEAD` / `KXPGAR2LEAD` / `KXPGAR3LEAD` | — | 66–117 | End-of-round leader |
| `KXPGATOP5` / `KXPGATOP10` / `KXPGATOP20` | `Top N Finishers` | 66–117 | Multi-pick |
| `KXPGAR1TOP5` / `KXPGAR1TOP10` | `Round 1 Top N Finishers` | 66–117 | After round 1 |
| `KXPGAMAKECUT` | `To Make the Cut` | 66–117 | Binary YES per golfer |
| `KXPGAHOLEINONE` | `Hole-in-One` | 3 | `1+ / 2+ / 3+ holes-in-one` thresholds |
| `KXPGAPLAYOFF` | `Playoff` | 1 | Binary YES |

**Tour variants** (parallel structure on smaller fields):
- `KXLIVTOUR` / `KXLIVR1LEAD` / `KXLIVTOP5` / `KXLIVTOP10` — LIV Golf, ~57 golfers
- `KXLIVOCCUR` — binary YES (will tournament happen)
- `KXDPWORLDTOUR` / `KXDPWORLDTOURR1LEAD` / `KXDPWORLDTOURMAKECUT` — DP World Tour, ~149–160 golfers (Europe)
- `KXLPGATOUR` — LPGA, ~119 women golfers

**Multi-tournament season-long futures:**
- `KXPGAMAJORWIN` (52 golfers — winner of any 2026 major)
- `KXPGAMAJORTOP10` (11 — top-10 in all 4 majors)
- `KXGOLFMAJOR` / `KXGOLFMAJORS` (per-player thresholds: `1+ / 2+ / 3+ / 4 majors`)
- `KXRYDERCUPCAPTAIN` (US + Europe captain)
- `KXPGARYDER` / `KXPGASOLHEIM` (3 outcomes: US / Europe / Tie)

**Per-player binary futures:** `KXPGACURRY`, `KXPGATIGER`, `KXSCOTTIESLAM`, `KXBRYSONCOURSERECORDS`.

**Cross-sport:** `KXGOLFTENNISMAJORS` (3 outcomes: golf player / tennis player / Tie).

### Universal: outrights / futures (no suffix)

Outcome count is variable (1 to 70+), labels are sport-specific:
- 1 outcome: binary YES/NO speculation (e.g., `Will Lamine Yamal leave Barcelona`)
- 2–48 team_like: champion / division / conference / playoff / draft / squad
- 10–72 player_like: MVP / ROY / Cy Young / Calder / Goal Leader / Top RB / etc.
- 16–225 pair labels: `Finals Matchup` (e.g., `PSG vs Atletico`)

### Universal facts
- **Field coverage = 100%** for `prob` / `yes` / `no` / `ticker` across every sport, every bucket. Kalshi reliably ships prices everywhere; missing prices in our UI are bugs in our code, not in their data.
- **Outcome label IS the team name** in `GAME` markets — no parsing needed.
- **Outcome label starts with team name** in `SPREAD` markets — extractable via prefix match.

## 5. Ticker grammar (per sport)

Source: `probe_tickers.py` → `/api/_debug/ticker_grammar`.

### G1 — date + teams (no time)

```
{base}{suffix}-{YYMMDD}{HOME_ABBR}{AWAY_ABBR}
```

Used by:
- **All Soccer** leagues (KXEPL, KXLALIGA, KXSERIEA, KXBUNDESLIGA, KXLIGUE1, KXMLS, KXJLEAGUE, KXKLEAGUE, KXSAUDIPL, KXCHNSL, KXBRASILEIRO, KXUCL, KXUEL, KXUECL, KXCONMEBOLLIB, KXCONMEBOLSUD, KXUAEPL, KXEGYPL, KXARGPREMDIV, KXNWSL, KXEWSL, KXBOLPDIV, KXURYPD, KXCONCACAFCCUP, KXWC)
- **NBA** (KXNBA + all sub-markets including player props)
- **NHL** (KXNHL + sub-markets)
- **UFL**

Examples:
- `KXEPLGAME-26MAY19CFCTOT` → date 2026-05-19, home CFC, away TOT
- `KXNBAGAME-26MAY11OKCLAL` → home OKC, away LAL
- `KXNHLOVERTIME-26MAY10VGKANA` → home VGK, away ANA
- `KXUFLGAME-26MAY10ORLHOU` → home ORL, away HOU

**Date format**: `YY` + 3-letter UPPERCASE month + `DD`. Locale-independent, UTC-anchored.

**Team block**: variable length 4–8 chars total. Parsed by **walking from the right** to the longest known team-abbr suffix, then home_abbr is what's left. Real-world cases:
- `OKCLAL` (3+3) — Oklahoma City + LA Lakers
- `LAFCHOU` (4+3) — LAFC + Houston
- `M05UNI` (3+3 with digit) — Mainz 05 + Union Berlin
- `PITAZ` (3+2) — Pittsburgh + Arizona

**Team abbr regex must allow digits**: `[A-Z0-9]+` (not `[A-Z]+`). Five MLS / Bundesliga tickers in our sample fail a strict-alpha regex.

### G7 — date + time + teams

```
{base}{suffix}-{YYMMDD}{HHMM}{HOME_ABBR}{AWAY_ABBR}
```

Used by:
- **All MLB** per-fixture markets (because doubleheaders)
- **KBO**, **NPB** (baseball)
- **AHL**, **SHL**, **KHL**, **DEL**, **Liiga** (international hockey)
- **EuroLeague**, **BBL**, **CBA**, **ISL**, **ABA**, **ARGLNB**, **FIBACHAMPLEAGUE**, **LNBELITE**, **BSL**, **JBLEAGUE**, **NZNBL**, **GBL**, **KBL**, **ACB** (international basketball)

Examples:
- `KXMLBGAME-26MAY071540PITAZ` → date 2026-05-07, time 15:40, home PIT, away AZ
- `KXEUROLEAGUEGAME-26MAY061415VALPAN` → time 14:15
- `KXAHLGAME-26MAY061900MANGRA` → time 19:00

**Time is part of the fixture identity** — same team-pair on same date can mean two different games (doubleheader).

### G3 — year (4-digit)
```
{base}-{YYYY}     e.g. KXPFAPOY-2526, KXPLAYTOGETHERJBJT-2027
```

### G4 — short year (2-digit)
```
{base}-{YY}       e.g. KXUCL-26, KXNBAMVP-26, KXSB-27
```

Used by virtually every outright / season-long / future market across all sports.

### G5 — handle
```
{base}-{HANDLE}   e.g. KX1STHOMEGAME-BUF26, KXNEXTNBACOACH-POR26
```

### G_SERIES — NBA / NHL playoff series (round-scoped)
```
{base}-{YY}{HOME_ABBR}{AWAY_ABBR}R{N}
```

Examples: `KXNBASERIES-26LALOKCR2`, `KXNHLSERIESSCORE-26MTLBUFR2`

`R{N}` is the playoff round number. No date — the market spans the full series.

### G_LEG — sub-fixture sub-markets (set/map/period leg)

Generalized pattern. Two variants depending on whether the parent grammar is G1 or G7:

```
G_LEG_DATE        — {base}-{YYMMDD}{ABBRS}-{N}              (Tennis)
G_LEG_DATE_TIME   — {base}-{YYMMDD}{HHMM}{ABBRS}-{N}        (Esports)
```

`N` is the leg number (1, 2, 3, …). The leg can be a tennis set, an esports map, or any future "leg of a series" market.

**Tennis examples** (no time):
- `KXATPSETWINNER-26MAY05HIJBAS-1` — Set 1
- `KXWTASETWINNER-26MAY05TOWJIM-2` — Set 2

**Esports examples** (with time):
- `KXCS2MAP-26MAY051800VSCBSTA-1` — Map 1
- `KXLOLMAP-26MAY071500ZYBSLY-2` — Map 2
- `KXVALORANTMAP-26MAY071400VITTL-1`
- `KXDOTA2MAP-26MAY051400L1GANEM-1`

The base ticker (without `-{N}`) matches the corresponding parent `MATCH` / `GAME` ticker, so a leg can be deterministically associated with its parent match.

### G_DATE_ONLY — date-keyed outright (Tennis)

```
{base}-{YYMMDD}
```

Used for "ranked / state on date X" type futures. Examples: `KXATP1RANK-26DEC31`, `KXATP1RANK-26JUN30`.

### Tennis player abbreviations

Tennis uses **first 3 chars of surname** as the player abbreviation, concatenated home+away → 6 chars total. Examples:
- `HIJBAS` → Hijikata + Basilashvili
- `TOWJIM` → Townsend + Jimenez Kasintseva
- `MILPAV` → Milovanova + Pavlova

**Edge case**: compound surnames can produce 7-char blocks. `KXATPCHALLENGERMATCH-26MAY05DIABER` → Diaz Acosta (4-char `DIA`) + Bernet (`BER`). Rare (1 in our sample).

**Edge case 2**: rematch / same-day-second-bracket. `KXITFWMATCH-26MAY04CHOCHO2` — same `CHOCHO` abbrs with trailing `2`. Rare (~0.75% of ITFW matches). Renderer should keep the `2` as part of fixture identity to avoid collisions.

### G_TOURNAMENT_HANDLE — Golf tournament-scoped sub-markets

```
{base}-{TOURNAMENT_CODE}{YY}
```

Tournament code is a 3–7 char alphabetic event identifier; year is 2-digit. Used by all per-tournament Golf markets.

Examples:
- `KXPGATOUR-PGC26` → PGA Championship 2026
- `KXPGATOUR-TRC26` → Truist Championship 2026
- `KXPGATOUR-ONMBC26` → ONEflight Myrtle Beach Classic 2026
- `KXLIVTOUR-LIGV26` → LIV Golf Virginia 2026
- `KXDPWORLDTOUR-ESDCC26` → Estrella Damm Catalunya Championship 2026
- `KXLPGATOUR-MIZAO26` → Mizuho Americas Open 2026

**Lesson**: identify a Golf tournament by `(sport=Golf, base, handle)`. All 12 sub-markets attach automatically — same insight as MMA's per-fight identity, but scoped to a tournament instead of a fixture.

### MMA fighter abbreviations — clean G1, all sub-markets share identity

MMA per-fight ticker uses identical `{YYMMDD}{fighter_abbrs}` across **all 6 series** (FIGHT + 5 sub-markets):

```
KXUFCFIGHT-26MAY16MGOMOR        ← headline
KXUFCDISTANCE-26MAY16MGOMOR     ← binary "goes the distance"
KXUFCROUNDS-26MAY16MGOMOR       ← when does the fight end
KXUFCVICROUND-26MAY16MGOMOR     ← who wins in which round
KXUFCMOV-26MAY16MGOMOR          ← method of victory (per-fighter)
KXUFCMOF-26MAY16MGOMOR          ← method of finish (fighter-agnostic)
```

**Lesson**: identify a fight by `(sport=MMA, date, abbrs)` — all 6 sub-markets attach automatically. Cleanest sub-market structure of any sport so far. Fighter abbr = first 3 chars of surname × 2 = 6-char block.

### Esports team abbreviations — opaque blocks

Esports team abbr blocks vary 5–8 chars with **no consistent home/away split**:
- `VSCBSTA` (Vasco Esports + BESTIA Academy) — could be 3+4 or 4+3
- `L1GANEM` (L1ga Team + Team Nemesis) — includes a digit
- `HRTSKOIA` (Heretics + KOI Academy?) — 8 chars
- `1WINMOUZ` (1win + MOUZ) — leading digit, 4+4

**Lesson for the merge contract**: for fixture identity, **treat the abbr block as opaque**. Don't attempt home/away split. Full date+time+block is the join key. Home/away orientation comes from outcome labels (which carry full team names) at render time, not from parsing the ticker. This rule generalizes across all sports — `OKCLAL` and `LALOKC` would normally need sorting to match, but if FL's tickers and Kalshi's tickers use the same author, equality on the raw block works directly.

### Sport-specific outright ticker variations

Some bases use unique encodings that don't fit G1–G7. They appear only in non-h2h markets, so they don't affect FL pairing — list provided for completeness:

- `KXMLBWINS-{TEAM_ABBR}-{YY}` — e.g. `KXMLBWINS-WSH-26` (per-team season wins)
- `KXNEXTTEAMNFL-{YY}{PLAYER}` — e.g. `KXNEXTTEAMNFL-27JALLEN`
- `KXSTARTINGQBWEEK1-W1-{YYMMDD}-{TEAM}` — `KXSTARTINGQBWEEK1-W1-26SEP15-PIT`

## 7. Merge contract: Kalshi ↔ FlashLive

**The deterministic rule that /sports v2 implements.**

### 7.1 Kalshi-side identity

For each Kalshi cache record `r`:

```
sport         = r._sport                                      (set at cache build)
series_base   = strip_known_suffix(r.series_ticker)
suffix        = the stripped suffix (or "" for no suffix)
fixture_kind  = classify(series_base, suffix)
                  ∈ { per_fixture, series, per_team, outright }
```

**If `fixture_kind == per_fixture`**:
```
parsed = parse_ticker(r.event_ticker, sport)   # G1 or G7 per §5

kalshi_identity = (
    sport,
    parsed.date,
    parsed.time,                  # None for G1, HHMM for G7
    frozenset({parsed.home_abbr, parsed.away_abbr}),  # order-agnostic
)
```

**If `fixture_kind == series`** (NBA / NHL playoff series):
```
parsed = parse_series_ticker(r.event_ticker)

series_identity = (
    sport,
    series_base,
    parsed.year,
    frozenset({parsed.home_abbr, parsed.away_abbr}),
    parsed.round,
)
```

**If `fixture_kind == per_team`** (KXMLBWINS, KXNFLWINS, etc.):
```
team_identity = (sport, series_base, year, team_abbr)
```

**If `fixture_kind == outright`** (champion, MVP, ROY, futures, …):
```
outright_identity = (sport, series_base, year)
```

### 7.2 FL-side identity (per FL event `e`)

```
fl_identity = (
    sport,                                       # FL sport_id → name
    e.START_TIME → date (UTC),
    e.START_TIME → time HHMM (UTC),
    frozenset({e.SHORTNAME_HOME, e.SHORTNAME_AWAY}),
)
```

### 7.3 The pairing rule

```
match(k, e) iff
    k.sport == e.sport
  AND k.kalshi_identity.date == e.fl_identity.date    (±1 day fuzz for tz drift)
  AND k.kalshi_identity.team_set == e.fl_identity.team_set
  AND IF k.kalshi_identity.time is not None:
        |k.time - e.fl_identity.time| <= 30 minutes   (kickoff drift tolerance)
```

This is a **deterministic equality** check on parsed identity tuples. Note what's *not* in the rule:
- ❌ no `match_game` fuzzy team-name search
- ❌ no `_kalshi_title_corroborates_fl_game` token gymnastics
- ❌ no second-pass attach pass
- ❌ no team-alias dictionary
- ❌ no nickname expansion

The complexity in the current code exists because we were inferring identity from titles + team names. With identity computed from tickers + SHORTNAMEs, all of that goes away.

### 7.4 Renderer rule

Once paired (or for stand-alone Kalshi-only events), the renderer
builds outcomes per the `(sport, suffix)` table in §4. No
shape-guessing from titles.

### 7.5 League bucket rule

Bucket assignment is keyed by `series_base` → FL `tournament_stage_id`,
learned from observed pairings (with cross-request persistence).
No SOCCER_COMP-style hand-curated map needed for bucketing
(it's still useful for **labels** when no FL fixture is paired,
but not for routing).

### 7.6 What this fixes

| Past bug | Why it happened | How the contract prevents it |
|---|---|---|
| Same fixture in two leagues | Synthesized bucket name ≠ FL bucket name | series_base routing keyed by observed pairings |
| Missing away team in card | Outcome-label parsing assumed shape | Per-(sport,suffix) outcome table |
| Wrong-leg pairing for cup ties | match_game returned earlier leg | Date is in the ticker — wrong leg can't match |
| Atl. Madrid vs Atletico no match | Token corroboration | Tickers use SHORTNAME `ATM` directly |
| Bayern Munich vs PSG no match | match_game phrase miss | SHORTNAMEs do the work, no phrase matching |
| Duplicates from suffix splits | matchup-key inconsistencies | Identity is parsed, not derived from title |
